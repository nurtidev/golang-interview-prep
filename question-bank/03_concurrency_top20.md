# Топ 20 вопросов по конкурентности

---

**1. В чём разница между конкурентностью и параллелизмом?**
> Конкурентность — структура программы (несколько задач в процессе). Параллелизм — физическое одновременное выполнение на нескольких CPU. Можно иметь конкурентность без параллелизма (1 ядро) и параллелизм без конкурентности (SIMD).

**2. Что такое race condition?**
> Ситуация когда результат зависит от порядка выполнения горутин. Обнаружение: `go test -race` или `go run -race`. Никогда не игнорировать.

**3. Как работает паттерн Fan-Out / Fan-In?**
> **Fan-Out** — один канал разделяется на N воркеров для параллельной обработки. **Fan-In** — N каналов объединяются в один результирующий. Используется для распараллеливания вычислений.

**4. Что такое Worker Pool и когда использовать?**
```go
jobs := make(chan Job, 100)
for i := 0; i < numWorkers; i++ {
    go worker(jobs)
}
```
> Ограниченный набор горутин обрабатывает очередь задач. Используется когда нельзя создавать неограниченное количество горутин (защита от перегрузки).

**5. Как реализовать rate limiter через канал?**
```go
limiter := time.Tick(200 * time.Millisecond)
for req := range requests {
    <-limiter
    go process(req)
}
```

**6. Что такое Pipeline паттерн?**
> Цепочка горутин, каждая принимает данные из канала, трансформирует и отправляет в следующий канал. Позволяет обрабатывать данные поэтапно с конкурентностью между этапами.

**7. Как правильно отменить группу горутин?**
> Через `context.WithCancel` или `context.WithTimeout`. Передавать context первым параметром. Горутины периодически проверяют `ctx.Done()`.

**8. Что такое semaphore паттерн в Go?**
```go
sem := make(chan struct{}, N) // буферизованный канал как семафор
sem <- struct{}{}             // acquire
defer func() { <-sem }()     // release
```
> Ограничивает количество одновременно выполняемых операций.

**9. Как gracefully завершить worker pool?**
> Закрыть канал задач `close(jobs)`. Workers выйдут из `range jobs` когда канал закроется и опустеет. Дождаться завершения через `sync.WaitGroup`.

**10. Что такое context cancellation propagation?**
> Отмена родительского контекста автоматически отменяет все дочерние. Создаёт дерево отмены. При HTTP запросе — разрыв соединения отменяет `r.Context()` и все дочерние.

**11. Как избежать goroutine leak при использовании каналов?**
> Всегда иметь exit condition: `select { case v := <-ch: ... case <-ctx.Done(): return }`. Закрывать каналы когда отправка завершена. Использовать `goleak` в тестах.

**12. Что такое `errgroup`?**
```go
g, ctx := errgroup.WithContext(ctx)
g.Go(func() error { return doWork(ctx) })
if err := g.Wait(); err != nil { ... }
```
> `golang.org/x/sync/errgroup` — группа горутин с обработкой ошибок. При ошибке одной горутины отменяет контекст остальных.

**13. Как работает `sync.Cond`?**
> Условная переменная. `Wait()` — атомарно освобождает mutex и ждёт сигнала. `Signal()` — будит одну горутину. `Broadcast()` — будит все. Используется для producer-consumer с условием.

**14. В чём разница между mutex и channel для синхронизации?**
> Mutex — для защиты разделяемого состояния (shared memory). Channel — для передачи данных между горутинами (communication). Правило Go: "не коммуницируй через shared memory, используй shared memory через communication".

**15. Что такое false sharing и как его избежать?**
> Две независимые переменные в одной cache line процессора — при изменении одной сбрасывается cache line у других ядер. Фикс: padding между переменными до размера cache line (64 байта).

**16. Что произойдёт если дважды вызвать `wg.Done()` больше чем `wg.Add()`?**
> `panic: sync: negative WaitGroup counter`. Счётчик WaitGroup не может быть отрицательным.

**17. Как работает `sync.Map.LoadOrStore`?**
> Атомарно: если ключ есть — возвращает существующее значение. Если нет — записывает и возвращает новое. Полезно для once-инициализации без дополнительных локов.

**18. Что такое happens-before в Go memory model?**
> Гарантия порядка видимости операций между горутинами. Отправка в канал happens-before получения. Закрытие канала happens-before получения zero value. Без синхронизации — нет гарантий порядка.

**19. Как запустить таймаут для горутины?**
```go
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()
select {
case result := <-doWork(ctx):
    // обработка
case <-ctx.Done():
    return ctx.Err()
}
```

**20. Что такое `time.After` и почему он может течь?**
> `time.After(d)` создаёт таймер и канал. Если select выходит по другому case — таймер не отменяется, горутина живёт до d. В hot loop использовать `time.NewTimer` с `timer.Stop()` + `timer.Reset()`.
