# Топ 100 вопросов по Go

---

## Слайсы (1–15)

**1. Из чего состоит слайс под капотом?**
> Три поля: `ptr` (указатель на базовый массив), `len` (длина), `cap` (ёмкость).

**2. Что происходит при append, если cap не исчерпан?**
> Новая аллокация не происходит — элемент записывается прямо в существующий базовый массив. len увеличивается на 1.

**3. Что происходит при append, если cap исчерпан?**
> Go аллоцирует новый массив большего размера, копирует все элементы, возвращает новый слайс с новым ptr.

**4. По какой формуле растёт cap при append?**
> До ~256 элементов — удваивается (x2). После 256 — увеличивается примерно на 25% (формула плавная, зависит от размера элемента, изменена в Go 1.18).

**5. Что выведет код?**
```go
a := []int{1, 2, 3}
b := a[:2]
b = append(b, 99)
fmt.Println(a)
```
> `[1 2 99]`. `b` и `a` смотрят на один базовый массив. append пишет 99 в `a[2]`, т.к. cap хватает.

**6. Как скопировать слайс так, чтобы изменения в копии не влияли на оригинал?**
```go
dst := make([]int, len(src))
copy(dst, src)
```

**7. Что вернёт `len` и `cap` для `var s []int`?**
> Оба вернут 0. nil-слайс валиден, append на него работает.

**8. Чем отличается `nil` слайс от пустого слайса `[]int{}`?**
> nil-слайс: ptr=nil, len=0, cap=0. Пустой слайс: ptr указывает на специальный адрес, len=0, cap=0. Оба работают с append и range. Разница только в `s == nil`.

**9. Как возникает утечка памяти через слайс?**
> Если взять подслайс большого слайса — весь базовый массив остаётся в памяти, пока жив подслайс. Фикс: `copy` в новый слайс нужного размера.

**10. Что делает трёхиндексный срез `a[1:3:4]`?**
> Ограничивает cap. Результат имеет len=2, cap=3 (4-1). Защищает от случайной записи за пределы среза через append.

**11. Можно ли сравнивать два слайса оператором `==`?**
> Нет, компилятор запрещает. Только с nil. Для сравнения используй `reflect.DeepEqual` или `slices.Equal` (Go 1.21+).

**12. Что быстрее: передать слайс в функцию или указатель на слайс?**
> Передача слайса уже передаёт заголовок (ptr+len+cap = 24 байта). Указатель на слайс нужен только если функция должна изменить сам заголовок (например, append внутри функции чтобы изменение было видно снаружи).

**13. Как удалить элемент по индексу из слайса?**
```go
s = append(s[:i], s[i+1:]...)
```

**14. Как работает `copy` если dst меньше src?**
> Копирует `min(len(dst), len(src))` элементов. Не паникует.

**15. Что такое `[]byte` и как преобразовать `string` в `[]byte` без аллокации?**
> Обычное преобразование `[]byte(s)` копирует данные. Без аллокации — через `unsafe`: `*(*[]byte)(unsafe.Pointer(&s))`, но это небезопасно. В Go 1.20+ есть `unsafe.SliceData`.

## Мапы (16–28)

**16. Как устроена map под капотом?**
> `hmap` структура: массив бакетов (`bmap`), каждый бакет хранит 8 пар ключ-значение + tophash (8 бит хэша для быстрого сравнения). При переполнении создаются overflow-бакеты.

**17. Что происходит при конкурентном чтении и записи в map?**
> Race condition — undefined behavior. В Go 1.6+ runtime обнаруживает конкурентную запись и делает `fatal error: concurrent map writes`.

**18. Как сделать потокобезопасную map?**
> Два варианта: `sync.Mutex` + обычная map, или `sync.Map`. `sync.Map` оптимальна для случая "много читателей, мало писателей" или когда ключи не пересекаются между горутинами.

**19. Когда использовать `sync.Map` вместо `map + Mutex`?**
> `sync.Map` выгоднее когда: (1) запись происходит один раз, чтение много раз; (2) горутины работают с разными ключами. В остальных случаях `map + RWMutex` обычно быстрее.

**20. Гарантирован ли порядок итерации по map?**
> Нет. Go намеренно рандомизирует порядок с Go 1.0, чтобы код не зависел от порядка.

**21. Что вернёт обращение к несуществующему ключу?**
> Zero value для типа значения. Проверка: `v, ok := m[key]` — ok будет false.

**22. Можно ли использовать слайс как ключ map?**
> Нет. Ключ должен быть comparable. Слайсы, map, функции — не comparable. Используй массив `[N]int` вместо слайса.

**23. Как реализовать Set в Go?**
```go
set := make(map[string]struct{})
set["key"] = struct{}{}
_, exists := set["key"]
```

**24. Что происходит с map при росте (rehashing)?**
> При load factor > 6.5 (в среднем) Go создаёт новый массив бакетов вдвое больше и постепенно (инкрементально) переносит данные. Во время rehash оба массива бакетов существуют.

**25. Можно ли удалять элементы из map во время итерации?**
> Да, это безопасно в Go. Удалённые элементы могут или не могут появиться в текущей итерации.

**26. Почему нельзя взять адрес элемента map?**
```go
m := map[string]int{"a": 1}
p := &m["a"] // ошибка компиляции
```
> Элементы map могут перемещаться в памяти при rehash, поэтому адрес был бы нестабилен.

**27. Как инициализировать map с начальной ёмкостью?**
```go
m := make(map[string]int, 100) // hint для аллокатора
```

**28. Чем отличается `map[K]V` от `map[K]*V`?**
> `map[K]*V` позволяет изменять поля структуры через указатель. В `map[K]V` нельзя писать `m[k].Field = x` — нужно копировать, изменять, класть обратно.

## Интерфейсы (29–40)

**29. Из чего состоит интерфейс под капотом?**
> `iface` = два указателя: `itab` (тип + таблица методов) и `data` (указатель на данные). Пустой интерфейс `interface{}` = `eface` (type + data).

**30. Что такое nil interface trap?**
```go
var err *MyError = nil
var iface error = err
fmt.Println(iface == nil) // false!
```
> Интерфейс nil только если оба поля (type и data) nil. Здесь type = *MyError, data = nil → интерфейс не nil.

**31. Как правильно вернуть nil ошибку через интерфейс?**
```go
func f() error {
    var err *MyError = nil
    return err // ПЛОХО — вернёт non-nil interface
}
func f() error {
    return nil // ХОРОШО — явный nil interface
}
```

**32. Что такое type assertion и type switch?**
```go
v, ok := i.(string)      // type assertion
switch v := i.(type) {   // type switch
case string: ...
case int: ...
}
```

**33. Чем отличается interface{} от any?**
> `any` — это алиас для `interface{}` добавленный в Go 1.18. Полностью идентичны.

**34. Можно ли сравнивать два interface{}?**
> Да, но паникует если динамический тип не comparable (например, слайс внутри interface{}).

**35. Что такое duck typing в Go?**
> Тип реализует интерфейс неявно — просто имея нужные методы. Не нужно явно писать `implements`.

**36. Как проверить что тип реализует интерфейс (compile-time)?**
```go
var _ MyInterface = (*MyStruct)(nil)
```

**37. Когда использовать интерфейс с одним методом vs конкретный тип?**
> Интерфейс — когда нужна подстановка (тестируемость, несколько реализаций). Конкретный тип — когда реализация одна и абстракция не нужна.

**38. Что такое embedding интерфейсов?**
```go
type ReadWriter interface {
    Reader
    Writer
}
```
> Интерфейс наследует методы вложенных интерфейсов.

**39. Влияет ли использование интерфейсов на производительность?**
> Да: вызов метода через интерфейс — indirect call через vtable (itab), что мешает инлайнингу. Также boxing значений в interface{} может вызывать heap allocation.

**40. Что такое `errors.Is` и `errors.As`?**
> `errors.Is` — проверяет цепочку ошибок на совпадение значения. `errors.As` — ищет в цепочке ошибку нужного типа и присваивает в target. Используют `Unwrap()` для обхода цепочки.

## Defer, Panic, Recover (41–50)

**41. В каком порядке выполняются несколько defer?**
> LIFO (Last In, First Out) — стек. Последний defer выполнится первым.

**42. Когда вычисляются аргументы defer?**
> В момент вызова defer, а не в момент выполнения. Результат функции — нет (захватывается по ссылке если именованный возврат).

**43. Что выведет код?**
```go
func f() (result int) {
    defer func() { result++ }()
    return 1
}
```
> `2`. defer может изменять именованные возвращаемые значения.

**44. Можно ли recover из паники в другой горутине?**
> Нет. recover работает только в той же горутине. Паника в горутине без recover убивает всю программу.

**45. Как правильно использовать recover?**
```go
defer func() {
    if r := recover(); r != nil {
        log.Println("recovered:", r)
    }
}()
```
> recover должен быть вызван непосредственно в defer-функции, а не во вложенной.

**46. Влияет ли defer на производительность?**
> Да, небольшой overhead. В Go 1.14 добавили open-coded defers — компилятор встраивает defer inline если нет циклов/условий, что почти убирает накладные расходы.

**47. Что произойдёт если panic случится в defer?**
> Паника продолжится, предыдущая паника будет потеряна (в Go 1.21+ можно видеть обе через `errors.Join`-подобный механизм).

**48. Зачем использовать defer для закрытия ресурсов?**
> Гарантирует закрытие даже при панике или раннем return. Код более читабелен — открытие и закрытие рядом.

**49. Что делает `runtime.Goexit()`?**
> Завершает текущую горутину, выполнив все defer. В отличие от panic, не распространяется.

**50. Какова разница между `panic` и `os.Exit`?**
> `panic` выполняет defer и может быть перехвачен через recover. `os.Exit` немедленно завершает программу, defer не выполняются.

## Горутины и GMP (51–63)

**51. Что такое горутина?**
> Легковесный поток управляемый Go runtime. Начальный стек ~2-8KB (растёт динамически до 1GB). Переключение контекста намного дешевле OS-потоков.

**52. Что такое G, M, P в GMP модели?**
> G — горутина. M — OS-поток (machine). P — логический процессор (контекст выполнения). M нужен P чтобы выполнять горутины.

**53. Сколько P создаётся по умолчанию?**
> `GOMAXPROCS` = количество логических CPU ядер. Можно изменить через `runtime.GOMAXPROCS()`.

**54. Что такое work stealing?**
> Когда P исчерпал свою локальную очередь горутин, он крадёт половину горутин из локальной очереди другого P (или берёт из глобальной очереди).

**55. Что происходит с P при блокирующем syscall?**
> P отсоединяется от M и привязывается к другому (или новому) M. Заблокированный M ждёт завершения syscall вместе с горутиной.

**56. Что такое preemption в Go?**
> Механизм принудительного вытеснения горутин. До Go 1.14 — кооперативное (только в точках вызова). С Go 1.14 — асинхронное preemption через сигналы (SIGURG).

**57. Как работает стек горутины?**
> Начинается с 2-8KB, растёт по необходимости (segmented stack до Go 1.3, затем copying stack). При росте создаётся новый больший стек, данные копируются.

**58. Что такое локальная и глобальная очередь горутин?**
> Каждый P имеет локальную очередь runq (размер 256). Глобальная очередь — для overflow и горутин от syscall. P периодически берёт горутины из глобальной (каждые 61 тик).

**59. Можно ли узнать ID текущей горутины?**
> Нет официального API. `runtime.Stack()` включает goroutine ID в stack trace, но это не рекомендуется использовать в продакшне.

**60. Что такое `runtime.Gosched()`?**
> Yield — передаёт управление планировщику, позволяя другим горутинам выполниться. Аналог `time.Sleep(0)`.

**61. Сколько максимально горутин можно создать?**
> Ограничено только памятью. Каждая горутина ~2-8KB стека. На практике десятки тысяч — норма, миллионы — возможны.

**62. Что такое goroutine leak?**
> Горутина создана но никогда не завершается (заблокирована навсегда на канале, мьютексе и т.д.). Накапливаются → утечка памяти.

**63. Как обнаружить утечку горутин?**
> `runtime.NumGoroutine()`, pprof endpoint `/debug/pprof/goroutine`, библиотека `goleak` (Uber) в тестах.

## Каналы (64–75)

**64. Чем отличается буферизованный канал от небуферизованного?**
> Небуферизованный блокирует отправителя до появления получателя (рандеву). Буферизованный блокирует только при полном буфере (отправку) или пустом (получение).

**65. Что происходит при чтении из закрытого канала?**
> Возвращает zero value и false: `v, ok := <-ch` → ok=false. Не паникует.

**66. Что происходит при записи в закрытый канал?**
> Паника: `send on closed channel`.

**67. Что происходит при закрытии уже закрытого канала?**
> Паника: `close of closed channel`.

**68. Что происходит при чтении из nil канала?**
> Блокируется навсегда.

**69. Что происходит при записи в nil канал?**
> Блокируется навсегда.

**70. Как работает select при нескольких готовых каналах?**
> Выбирает случайный (псевдослучайный) case. Это предотвращает starvation.

**71. Что делает `default` в select?**
> Выполняется если ни один канал не готов (неблокирующий select).

**72. Как правильно сигнализировать о завершении через канал?**
```go
done := make(chan struct{})
close(done) // broadcast всем получателям
```

**73. Кто должен закрывать канал — отправитель или получатель?**
> Всегда отправитель (или координатор). Получатель не знает придут ли ещё данные.

**74. Что такое однонаправленные каналы?**
```go
chan<- T  // только запись
<-chan T  // только чтение
```
> Используются в сигнатурах функций для ограничения доступа.

**75. Как реализовать таймаут через select?**
```go
select {
case result := <-ch:
    // обработка
case <-time.After(5 * time.Second):
    // таймаут
}
```

## Sync Primitives (76–83)

**76. Чем отличается Mutex от RWMutex?**
> `Mutex` — эксклюзивный lock (один поток). `RWMutex` — множественные читатели или один писатель. При наличии активных читателей `Lock()` ждёт их завершения.

**77. Что такое deadlock и как его избежать?**
> Две горутины ждут ресурсы друг друга. Избежать: всегда захватывать локи в одном порядке, использовать таймауты, минимизировать scope лока.

**78. Что такое `sync.Once`?**
> Гарантирует однократное выполнение функции. Потокобезопасно. Используется для ленивой инициализации синглтона.

**79. Как работает `sync.WaitGroup`?**
> `Add(n)` — увеличивает счётчик. `Done()` — уменьшает (вызывается в горутине). `Wait()` — блокирует до нуля счётчика.

**80. Что такое `sync.Pool`?**
> Пул временных объектов для переиспользования, снижает давление на GC. Объекты могут быть удалены GC в любой момент. Не для долгосрочного хранения.

**81. Что такое atomic операции и зачем они?**
> Неделимые операции на уровне CPU (CAS, Add, Load, Store). Быстрее mutex для простых счётчиков/флагов. `sync/atomic` пакет.

**82. Что такое CAS (Compare-And-Swap)?**
```go
atomic.CompareAndSwapInt64(&val, old, new)
```
> Атомарно: если val==old, устанавливает new и возвращает true. Основа lock-free алгоритмов.

**83. Чем опасен `sync.Mutex` при копировании?**
> Нельзя копировать после первого использования — копия содержит состояние оригинала. `go vet` обнаруживает `sync.Mutex` копирование.

## Memory и GC (84–92)

**84. Как Go решает — стек или куча для переменной?**
> Escape analysis: если переменная "уходит" за пределы функции (возвращается указатель, передаётся в интерфейс, захватывается замыканием) — аллоцируется на куче. Иначе — на стеке.

**85. Как работает GC в Go?**
> Tri-color mark-and-sweep. Три фазы: STW mark setup → concurrent marking (tri-color) → STW mark termination → concurrent sweep. Большую часть работы делает конкурентно.

**86. Что такое GOGC?**
> Управляет агрессивностью GC. `GOGC=100` (default) — GC запускается когда heap вырос на 100% относительно предыдущей коллекции. `GOGC=off` — отключить.

**87. Что такое GOMEMLIMIT (Go 1.19+)?**
> Мягкий лимит памяти. GC становится агрессивнее при приближении к лимиту. Предотвращает OOM kill.

**88. Что такое write barrier в GC?**
> Механизм отслеживания изменений указателей во время concurrent marking. Гарантирует что GC не пропустит живые объекты изменённые параллельно с маркировкой.

**89. Как уменьшить давление на GC?**
> `sync.Pool` для переиспользования объектов, предаллокация слайсов/map с нужной ёмкостью, избегать боксинга в interface{}, использовать `[]byte` буферы.

**90. Что такое escape analysis и как посмотреть результаты?**
```bash
go build -gcflags="-m" ./...
```
> Компилятор печатает: "variable X escapes to heap" или "does not escape".

**91. Чем отличается стек горутины от стека OS-потока?**
> Стек горутины начинается с 2-8KB и растёт динамически (copying). Стек OS-потока фиксирован (обычно 1-8MB). Поэтому можно создать миллионы горутин.

**92. Что такое finalizer и когда использовать?**
> `runtime.SetFinalizer(obj, func)` — вызывается GC перед удалением объекта. Не гарантирован по времени. Используется редко (для C ресурсов).

## Ошибки и общее (93–100)

**93. Как создать кастомный тип ошибки?**
```go
type NotFoundError struct {
    ID int
}
func (e *NotFoundError) Error() string {
    return fmt.Sprintf("not found: %d", e.ID)
}
```

**94. Чем `fmt.Errorf("... %w", err)` отличается от `fmt.Errorf("... %v", err)`?**
> `%w` оборачивает ошибку — `errors.Is/As` могут раскрутить цепочку. `%v` только форматирует строку, цепочка теряется.

**95. Что такое generics в Go (Go 1.18+)?**
```go
func Map[T, U any](s []T, f func(T) U) []U { ... }
```
> Параметрический полиморфизм. Типовые параметры в `[]`. Ограничения через interface.

**96. Что такое замыкание и классическая ловушка с горутинами?**
```go
for i := 0; i < 3; i++ {
    go func() { fmt.Println(i) }() // ПЛОХО: все горутины видят одну переменную i
}
// Фикс:
for i := 0; i < 3; i++ {
    i := i // создаём новую переменную
    go func() { fmt.Println(i) }()
}
```
> В Go 1.22+ переменная цикла for создаётся заново на каждой итерации — ловушка устранена.

**97. Что такое `init()` функция?**
> Выполняется автоматически при инициализации пакета. Может быть несколько в одном файле. Не может быть вызвана явно. Выполняется после инициализации всех зависимостей пакета.

**98. Что такое blank identifier `_`?**
> Выбрасывает значение. Используется для игнорирования возвращаемых значений, side-effect импортов (`import _ "pkg"`), compile-time проверок интерфейса.

**99. Чем `make` отличается от `new`?**
> `make` — только для slice, map, chan. Инициализирует внутреннее состояние. `new` — для любого типа, возвращает указатель на zero value, не инициализирует.

**100. Что такое `context.WithValue` и когда его использовать?**
> Передаёт значения через цепочку вызовов. Использовать только для request-scoped данных (trace ID, user ID). Не для опциональных параметров функций — это антипаттерн.
