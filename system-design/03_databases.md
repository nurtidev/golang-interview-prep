# Урок 3: Хранение данных

## Виды баз данных

Выбор типа БД — одно из первых архитектурных решений. Нет универсальной базы на все случаи жизни: каждый тип оптимизирован под свой паттерн доступа.

**Реляционные (PostgreSQL, MySQL)** — данные в таблицах со схемой, ACID-транзакции, JOIN-ы. Хороши для структурированных данных с отношениями. Масштабируются вертикально легко, горизонтально — сложно.

**Документоориентированные (MongoDB, Couchbase)** — данные в JSON-документах, гибкая схема. Хороши когда структура данных часто меняется или у разных записей разные поля. JOIN-ы — через денормализацию или несколько запросов.

**Wide-column (Cassandra, HBase)** — строки с динамическим набором колонок, распределённые по узлам. Оптимизированы под write-heavy нагрузку и хранение огромных объёмов. Нет JOIN-ов, ограниченные транзакции.

**Time-series (InfluxDB, TimescaleDB, VictoriaMetrics)** — специализированы под данные с временной меткой. Автоматическое сжатие, downsampling, функции по времени. Метрики, IoT, финансовые данные.

**Object Storage (S3, GCS)** — хранение бинарных объектов (файлы, фото, видео). Безлимитный объём, дёшево, но высокая latency. Не база данных в классическом смысле — нет запросов по содержимому.

| | Реляционная | Документная | Wide-column | Time-series |
|---|---|---|---|---|
| Схема | строгая | гибкая | гибкая | фиксированная |
| JOIN | да | нет | нет | нет |
| Транзакции | ACID | частично | нет / ограничены | нет |
| Масштабирование | вертикальное | горизонтальное | горизонтальное | горизонтальное |
| Примеры | PostgreSQL | MongoDB | Cassandra | InfluxDB |

## ACID vs BASE

**ACID** — гарантии реляционных БД, критичные для финансов и бизнес-логики:
- **Atomicity** — транзакция либо выполняется целиком, либо не выполняется вовсе
- **Consistency** — после транзакции БД переходит из одного валидного состояния в другое
- **Isolation** — параллельные транзакции не видят промежуточные состояния друг друга
- **Durability** — закоммиченные данные не теряются даже при сбое

**BASE** — модель распределённых NoSQL-систем, жертвующая строгостью ради доступности:
- **Basically Available** — система всегда отвечает, даже если данные неактуальны
- **Soft state** — состояние может меняться со временем (данные "доезжают")
- **Eventually consistent** — в конечном счёте все реплики придут к одному значению

> ACID vs BASE — это не выбор "лучше/хуже", это трейдофф: строгость vs масштабируемость.

## OLTP vs OLAP vs HTAP

**OLTP (Online Transaction Processing)** — короткие транзакции, много мелких операций: вставки, обновления, точечные чтения. Типичная продакшн-база: PostgreSQL, MySQL.

**OLAP (Online Analytical Processing)** — аналитические запросы по большим объёмам: агрегации, сканирование миллионов строк. Колоночное хранение эффективно для этого: Clickhouse, Redshift, BigQuery.

**HTAP (Hybrid)** — попытка совместить OLTP и OLAP в одной системе: TiDB, SingleStore. Сложно, но избавляет от ETL-пайплайна.

```
OLTP: SELECT * FROM orders WHERE id = 42          ← точечный запрос, 1 строка
OLAP: SELECT city, AVG(amount) FROM orders        ← скан всей таблицы,
      GROUP BY city WHERE year = 2024               агрегация миллионов строк
```

## Нормализация и денормализация

Нормализация — это процесс организации таблиц так, чтобы устранить дублирование данных и аномалии обновления. Данные разбиваются на связанные таблицы, связанные через внешние ключи.

**Аномалии без нормализации:**
```
orders: { order_id, product_name, product_price, customer_name, customer_email }

Проблемы:
- Изменился email клиента → нужно обновить все его заказы
- Удалили последний заказ клиента → потеряли данные о клиенте
- Новый клиент без заказа → нельзя сохранить (нет order_id)
```

**После нормализации:**
```
customers: { customer_id, name, email }
products:  { product_id, name, price }
orders:    { order_id, customer_id, product_id, quantity, created_at }
```

Нормальные формы (1NF → 2NF → 3NF) — формальные правила, но на практике достаточно понимать суть: **каждый факт хранится в одном месте**.

### Денормализация

Денормализация — намеренное введение дублирования ради производительности. Когда JOIN по нескольким таблицам становится узким местом — дублируем нужные поля в одну таблицу.

```sql
-- Нормализованный запрос (медленный при миллионах строк):
SELECT o.id, c.name, c.email, p.name, p.price
FROM orders o
JOIN customers c ON o.customer_id = c.id
JOIN products p ON o.product_id = p.id

-- Денормализованная таблица order_details (быстрый):
SELECT id, customer_name, customer_email, product_name, product_price
FROM order_details   ← всё в одной таблице, JOIN не нужен
```

**Цена денормализации:** при изменении данных нужно обновить все копии. Если email клиента изменился — нужно обновить и `customers`, и все строки в `order_details`. Решается через CDC или триггеры.

| | Нормализация | Денормализация |
|---|---|---|
| Дублирование | нет | есть |
| Скорость чтения | медленнее (JOIN) | быстрее (одна таблица) |
| Скорость записи | быстрее | медленнее (обновить копии) |
| Консистентность | проще поддерживать | сложнее |
| Когда | OLTP, частые обновления | OLAP, read-heavy, NoSQL |

> Нормализация — по умолчанию. Денормализация — осознанный трейдофф когда есть измеримая проблема с производительностью.

## Индексы

Индекс — это вспомогательная структура данных, которая ускоряет поиск ценой дополнительного места и замедления записи (индекс тоже нужно обновлять).

### Как работает BTree-индекс под капотом

BTree (Balanced Tree) — сбалансированное дерево поиска, где каждый узел содержит отсортированные ключи и указатели на дочерние узлы. Высота дерева маленькая — для таблицы в 1 миллиард строк она около 30 уровней.

```
Таблица users (1M строк), индекс на поле age:

                    [30 | 60]                   ← корневой узел
                   /    |    \
          [10|20|30] [40|50|60] [70|80|90]      ← внутренние узлы
          /   |   \       ...
    [ptr] [ptr] [ptr]                           ← листовые узлы (указатели на строки)

SELECT * FROM users WHERE age = 45:
  1. Читаем корень: 45 > 30 и 45 < 60 → идём в средний узел
  2. Читаем узел [40|50|60]: 45 > 40 и 45 < 50 → идём вниз
  3. Листовой узел: нашли pointer → читаем строку из heap
  Итого: 3-4 чтения с диска вместо full scan (1M чтений)
```

**BTree+ (используется в PostgreSQL, MySQL):** листовые узлы связаны в двусвязный список. Это позволяет эффективно выполнять range-запросы (`BETWEEN`, `>`, `<`) — нашли первый элемент, дальше идём по листьям линейно.

```
Range query: WHERE age BETWEEN 25 AND 35
  1. BTree-поиск: найти первый узел где age=25 → O(log N)
  2. Линейный обход по листьям: 25 → 26 → 27 → ... → 35 → O(K)
  где K — количество результатов
```

### Как работает Hash-индекс под капотом

Hash-индекс вычисляет `hash(key)` и хранит `{ hash → pointer_to_row }` в хеш-таблице. Поиск за O(1) — идеально для `=`. Но диапазоны невозможны: `hash(25)` и `hash(26)` никак не связаны.

```
SELECT * FROM users WHERE id = 42:
  hash(42) = 0x7F3A → bucket[0x7F3A] → pointer → строка
  Одно обращение к хеш-таблице + одно к данным
```

### Что происходит при вставке строки с индексом

Каждая `INSERT`/`UPDATE`/`DELETE` требует обновить все индексы таблицы — это цена за скорость чтения:

```
INSERT INTO users(name, age, email) VALUES ('Alice', 25, 'a@b.com'):
  1. Записать строку в heap (таблицу)
  2. Найти место в BTree-индексе на age=25 → вставить ключ
  3. Если листовой узел переполнен → split узла → обновить родителя
  4. То же для индекса на email
  → чем больше индексов, тем медленнее запись
```

### Типы индексов

**BTree** — универсальный. Поддерживает `=`, `<`, `>`, `BETWEEN`, `ORDER BY`. Работает для большинства задач.

**Hash** — только для `=`, но быстрее BTree для точных совпадений. Нельзя использовать для диапазонов.

**Bitmap** — для колонок с малым количеством уникальных значений (пол, статус). Эффективен для OLAP-запросов с `AND/OR`.

**GIN (Generalized Inverted Index)** — для полнотекстового поиска и массивов в PostgreSQL.

### Кластерный vs некластерный

**Кластерный индекс** — физический порядок строк на диске совпадает с порядком индекса. В таблице может быть только один. В PostgreSQL — это heap + отдельный индекс, в MySQL InnoDB — первичный ключ всегда кластерный.

**Некластерный** — индекс хранит указатели на строки. При чтении нужен дополнительный lookup в таблицу.

**Covering index (покрывающий)** — индекс содержит все колонки, нужные для запроса. Чтение только из индекса, без обращения к таблице:
```sql
-- Запрос: SELECT email FROM users WHERE city = 'Moscow'
-- Covering index: (city, email) — оба поля в индексе, таблица не нужна
CREATE INDEX idx_city_email ON users(city, email);
```

### Составной индекс и его порядок

В составном индексе порядок колонок критичен. Индекс `(a, b, c)` используется для запросов по `a`, `(a, b)`, `(a, b, c)`, но **не** для `(b, c)`.

```sql
-- Индекс (last_name, first_name)
WHERE last_name = 'Smith'              -- ✅ использует индекс
WHERE last_name = 'Smith' AND first_name = 'John'  -- ✅
WHERE first_name = 'John'             -- ❌ не использует (нет last_name)
```

## Уровни изоляции транзакций

Изоляция — это то, насколько параллельные транзакции видят изменения друг друга. Чем выше изоляция, тем ниже параллелизм и производительность.

| Уровень | Dirty Read | Non-Repeatable Read | Phantom Read |
|---------|------------|---------------------|--------------|
| READ UNCOMMITTED | возможен | возможен | возможен |
| READ COMMITTED | ✅ нет | возможен | возможен |
| REPEATABLE READ | ✅ нет | ✅ нет | возможен |
| SERIALIZABLE | ✅ нет | ✅ нет | ✅ нет |

**Dirty Read** — читаем данные незакоммиченной транзакции (которая может откатиться).

**Non-Repeatable Read** — один и тот же запрос в рамках транзакции возвращает разные данные (другая транзакция изменила строку между запросами).

**Phantom Read** — повторный запрос возвращает разное количество строк (другая транзакция вставила/удалила строки).

> Дефолт в PostgreSQL — READ COMMITTED, в MySQL InnoDB — REPEATABLE READ.

## Очереди сообщений

Очередь — это буфер между производителем и потребителем. Позволяет развязать скорость записи и обработки, сглаживать пики, делать retry при сбоях.

**Сценарии использования:**
- Асинхронная обработка: пользователь загрузил видео → очередь → воркер кодирует
- Развязка сервисов: сервис заказов публикует событие, сервис нотификаций подписывается
- Сглаживание пиков: 100k RPS → очередь → 10k RPS обработчиков

**Гарантии доставки:**
- **At most once** — сообщение доставляется 0 или 1 раз. Возможна потеря, зато дубликатов нет.
- **At least once** — сообщение доставляется 1 или более раз. Потерь нет, но нужна идемпотентность обработчика.
- **Exactly once** — ровно один раз. Дорого и сложно, требует транзакционности на стороне брокера.

```
Kafka:     at least once (по умолчанию), exactly once — через транзакции
RabbitMQ:  at least once
SQS:       at least once (standard), exactly once (FIFO queue)
```

## Materialized View и Stored Procedures

**Materialized View** — это результат запроса, сохранённый как таблица. В отличие от обычного view (который пересчитывается при каждом запросе), materialized view хранит данные физически и обновляется по расписанию или вручную. Полезно для дорогих OLAP-агрегаций.

```sql
CREATE MATERIALIZED VIEW daily_stats AS
SELECT date, COUNT(*), SUM(amount)
FROM orders GROUP BY date;

REFRESH MATERIALIZED VIEW daily_stats;  -- обновление
```

**Stored Procedure** — логика на стороне БД. Плюс: экономит сетевые round-trip. Минус: сложно тестировать, версионировать, деплоить. В микросервисной архитектуре — антипаттерн, логику держат в сервисах.

## Вопросы с собеса

**Когда выбирать NoSQL вместо реляционной БД?**

<details>
<summary>Ответ</summary>

NoSQL выбираем когда:
- **Гибкая схема нужна по требованиям** — поля у разных объектов разные (MongoDB)
- **Горизонтальное масштабирование write-heavy нагрузки** — Cassandra для миллионов событий/сек
- **Специализированные паттерны доступа** — time-series метрики, граф отношений, полнотекстовый поиск
- **Не нужны сложные JOIN и транзакции** — данные денормализованы под конкретные запросы

Реляционная БД остаётся выбором по умолчанию для бизнес-логики с транзакциями. Часто в одной системе используют обе: PostgreSQL для основных данных + Redis для кэша + Elasticsearch для поиска.

</details>

**Что такое Non-Repeatable Read и как его предотвратить?**

<details>
<summary>Ответ</summary>

Non-Repeatable Read — в рамках одной транзакции один и тот же SELECT дважды возвращает разные данные, потому что между запросами другая транзакция изменила и закоммитила строку.

```
T1: SELECT balance FROM accounts WHERE id=1  → 1000
T2: UPDATE accounts SET balance=500 WHERE id=1; COMMIT
T1: SELECT balance FROM accounts WHERE id=1  → 500  ← другой результат!
```

Предотвращается уровнем изоляции **REPEATABLE READ** и выше. PostgreSQL реализует это через MVCC (Multi-Version Concurrency Control) — транзакция видит снимок данных на момент своего начала.

</details>

---

**Почему порядок колонок в составном индексе важен?**

<details>
<summary>Ответ</summary>

BTree-индекс отсортирован по первой колонке, затем по второй внутри первой, и т.д. Использовать индекс можно только "с начала" — нельзя пропустить первую колонку.

Правило **"leftmost prefix"**: индекс `(a, b, c)` используется если в WHERE есть `a`, `(a, b)` или `(a, b, c)`. Запрос только по `b` или `(b, c)` индекс не использует.

При проектировании: первой ставим колонку с наибольшей селективностью (много уникальных значений) — это сужает выборку быстрее всего. Исключение: если запрос всегда фильтрует по одному фиксированному значению первой колонки (например, `status = 'active'`), её ставят первой даже при низкой селективности.

</details>

**В чём разница между нормализацией и денормализацией, когда что применять?**

<details>
<summary>Ответ</summary>

**Нормализация** — устраняем дублирование, каждый факт в одном месте. Упрощает запись и обновление, но чтение требует JOIN-ов.

**Денормализация** — намеренно дублируем данные ради скорости чтения. Ценой: при обновлении нужно синхронизировать все копии.

**Когда нормализуем:** OLTP-системы с частыми обновлениями (интернет-магазин, CRM). Целостность данных важнее скорости чтения.

**Когда денормализуем:**
- OLAP / аналитика: JOIN миллионов строк медленный, лучше одна широкая таблица
- NoSQL (Cassandra, MongoDB): JOIN-ов нет, данные моделируем под запрос
- Кэш / поисковый индекс: заранее вычисляем нужную проекцию данных

На практике: начинаем нормализованно, денормализуем конкретные узкие места после профилирования.

</details>

**Почему BTree-индекс не помогает для LIKE '%word%', но помогает для LIKE 'word%'?**

<details>
<summary>Ответ</summary>

BTree хранит ключи в отсортированном порядке и эффективен для поиска с известным **левым** краем диапазона.

`LIKE 'word%'` — ищем строки, начинающиеся с 'word'. В BTree это range-запрос: найти первый ключ ≥ 'word', идти по листьям пока ключ начинается с 'word'. Индекс используется.

`LIKE '%word%'` — ищем подстроку в любой позиции. Левый край неизвестен, отсортированность бесполезна — нужен full scan всего индекса. Для таких запросов используют **GIN/GiST индексы** с `pg_trgm` (триграммы) в PostgreSQL или полноценный Elasticsearch.

```sql
-- Триграммный индекс для PostgreSQL:
CREATE EXTENSION pg_trgm;
CREATE INDEX idx_trgm ON products USING GIN(name gin_trgm_ops);
-- Теперь LIKE '%word%' использует индекс
```

</details>

**Что такое at-least-once доставка и почему это требует идемпотентности?**

<details>
<summary>Ответ</summary>

At-least-once означает: брокер гарантирует, что сообщение будет доставлено хотя бы один раз. При сбоях (сеть, краш воркера до коммита offset) сообщение может прийти дважды.

Если обработчик не идемпотентен — дубликат вызовет повторное действие: пользователь дважды получит email, дважды спишутся деньги.

**Решения:**
- Хранить обработанные `message_id` в БД и проверять перед обработкой
- Делать операции naturally идемпотентными: `INSERT ON CONFLICT DO NOTHING`, `SET balance = X` вместо `balance = balance - Y`

</details>
