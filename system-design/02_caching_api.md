# Урок 2: Кэширование, API и Observability

## Кэширование

Кэш — это быстрое хранилище, которое держит копию часто запрашиваемых данных, чтобы не идти за ними каждый раз в медленный источник (БД, внешний сервис). Правильный кэш снижает latency в 10–100 раз и уменьшает нагрузку на базу.

**Внутренний кэш** живёт в памяти процесса — доступ за наносекунды, но данные не переживают рестарт и не видны другим инстансам.

**Внешний кэш** (Redis, Memcached) — отдельный сервис, общий для всех инстансов. Чуть медленнее (сетевой вызов ~1 мс), зато данные консистентны между серверами.

```
Запрос → App Server → [Redis Cache] → hit: ответ сразу
                              ↓ miss
                           [Database] → запись в кэш → ответ
```

---

## Алгоритмы вытеснения (Eviction)

Когда кэш заполнен, нужно что-то удалять. Выбор алгоритма зависит от паттерна доступа.

| Алгоритм | Логика | Когда использовать |
|----------|--------|--------------------|
| **LRU** (Least Recently Used) | Удаляем давно не используемый | Общий случай, популярно |
| **LFU** (Least Frequently Used) | Удаляем редко запрашиваемый | Стабильные горячие данные |
| **2Q** | LRU + буфер "пробных" данных | Защита от scan-pollution |
| **FIFO** | Удаляем первый вошедший | Потоковые данные, простота |
| **MRU** | Удаляем самый свежий | Рекурсивные паттерны доступа |

> **Проблема scan-pollution:** если одна большая выборка (full scan) прогоняет через кэш миллион записей — она вытолкнет все "горячие" данные. LRU уязвим к этому, 2Q и SLRU — нет.

---

## Паттерны взаимодействия с кэшем

**Cache-Aside (Lazy Loading)** — самый распространённый паттерн. Приложение само управляет кэшем: сначала смотрит в кэш, при miss — идёт в БД и записывает результат в кэш.

```
read:  cache.Get(key) → miss → db.Get(key) → cache.Set(key, val)
write: db.Update(key, val) → cache.Delete(key)  ← инвалидируем, не обновляем
```

**Write-Through** — при записи данные попадают и в кэш, и в БД одновременно. Кэш всегда актуален, но запись медленнее.

**Write-Behind (Write-Back)** — пишем в кэш немедленно, в БД — асинхронно. Максимальная скорость записи, но риск потери данных при падении кэша.

**Read-Through** — кэш сам умеет ходить в БД при miss. Приложение общается только с кэшем.

## Инвалидация кэша

Инвалидация — одна из двух сложных задач в программировании. Есть три подхода:

**TTL (Time-To-Live)** — данные протухают через фиксированное время. Просто, но данные могут быть устаревшими весь TTL.

**Event-based** — при изменении данных в БД явно удаляем или обновляем запись в кэше. Актуально, но сложнее — нужно не забыть инвалидировать везде.

**Cache Versioning** — добавляем версию в ключ: `user:42:v3`. При изменении данных меняем версию — старые ключи автоматически становятся "мёртвыми" и вытесняются по TTL.

## Thundering Herd Problem

Представь: кэш протух, и в этот момент приходят 10 000 одновременных запросов. Все идут в БД — база падает под нагрузкой.

```
Кэш протух → 10 000 запросов → 10 000 запросов в БД → 💥
```

**Решения:**
- **Mutex/Single-flight** — только один запрос идёт в БД, остальные ждут его результата
- **Probabilistic Early Expiration** — немного раньше TTL начинаем вероятностно обновлять кэш
- **Jitter к TTL** — `TTL = base + random(0, base*0.1)` — разбиваем одновременное протухание

## API стили

Выбор стиля API — это баланс между гибкостью, производительностью и удобством клиента.

| | REST | RPC (gRPC) | GraphQL |
|---|---|---|---|
| Формат | JSON/XML | Protobuf (бинарный) | JSON |
| Типизация | нет | строгая (.proto) | схема |
| Гибкость запроса | фиксированные endpoints | фиксированные методы | клиент выбирает поля |
| Производительность | средняя | высокая | средняя (N+1 проблема) |
| Стриминг | через SSE/WS | встроен | subscriptions |
| Когда | публичный API, CRUD | internal микросервисы | mobile, сложные UI |

**Under-fetching** — один запрос возвращает мало данных, нужно делать N дополнительных. GraphQL решает это напрямую.

**Over-fetching** — запрос возвращает лишние поля, которые клиент не использует. Тратим трафик и CPU на сериализацию.

## Polling, Long Polling, SSE, WebSocket

Когда клиенту нужны обновления в реальном времени — простой REST не подходит. Есть несколько подходов с разными трейдоффами.

**Polling** — клиент спрашивает сервер каждые N секунд: "есть что-то новое?". Просто, но неэффективно: большинство запросов получает пустой ответ.

**Long Polling** — клиент делает запрос, сервер держит соединение открытым до появления данных (или таймаута). Более эффективно, но держит поток/соединение на сервере.

**SSE (Server-Sent Events)** — однонаправленный стрим от сервера к клиенту по обычному HTTP. Прост в реализации, автоматическое переподключение встроено в браузер.

**WebSocket** — двунаправленное соединение. Клиент и сервер могут отправлять сообщения в любой момент.

```
Polling:      Client → req → Server (empty) → Client → req → Server (data!)
Long Polling: Client → req → Server ........... → (data появились) → ответ
SSE:          Client → req → Server → data → data → data → ...
WebSocket:    Client ⇄ Server (full-duplex, любое направление)
```

| | Polling | Long Polling | SSE | WebSocket |
|---|---|---|---|---|
| Направление | → | → | ← | ⇄ |
| Протокол | HTTP | HTTP | HTTP | WS |
| Сложность | низкая | средняя | низкая | высокая |
| Когда | редкие обновления | уведомления | лента событий | чат, игры |

## Идемпотентность

Операция идемпотентна, если её можно выполнить несколько раз — результат будет тот же, что и после первого раза. Критично для retry-логики: если запрос упал по таймауту, мы не знаем — дошёл он или нет. Безопасно повторить можно только идемпотентный запрос.

```
GET, PUT, DELETE — идемпотентны
POST — НЕ идемпотентен (создаёт новый ресурс каждый раз)
```

**Идемпотентный POST:** клиент генерирует `Idempotency-Key: uuid` и отправляет в заголовке. Сервер сохраняет результат по ключу — повторный запрос возвращает закешированный ответ, не выполняя операцию снова.

## Observability

Observability — это способность понять, что происходит внутри системы, глядя на её выходные данные. Три столпа: метрики, логи, трейсы.

**Метрики** — числа, агрегированные по времени. Latency, RPS, error rate, CPU usage. Хранятся в time-series БД (Prometheus, VictoriaMetrics). Дёшевы в хранении, хорошо для алертинга.

**Логи** — текстовые события с контекстом. Дороги в хранении и поиске, но незаменимы для дебага конкретного инцидента. Elasticsearch, Loki.

**Трейсы** — путь одного запроса через все микросервисы. Позволяют найти где именно возникает задержка в цепочке из 10 сервисов. Jaeger, Zipkin, OpenTelemetry.

```
Запрос → [Service A: 5ms] → [Service B: 120ms] → [DB: 95ms] → ответ
                                        ↑
                         Трейс показывает: узкое место в Service B
```

**Continuous Profiling** — постоянная запись CPU/memory профилей в продакшене. Позволяет найти "горячие" функции без ручного профилирования. Pyroscope, Parca.

## Версионирование API

API меняется, но старые клиенты нельзя сломать. Есть три подхода:

- **URL версионирование:** `/api/v1/users` → `/api/v2/users` — просто и явно, но загрязняет URL
- **Header версионирование:** `Accept: application/vnd.api+json;version=2` — чище, но менее заметно
- **Query param:** `/api/users?version=2` — плохо: кэши не различают версии

## Вопросы с собеса

**Что такое Thundering Herd и как его предотвратить?**

<details>
<summary>Ответ</summary>

Thundering Herd — когда истекает TTL популярного ключа в кэше, и все одновременные запросы идут в БД, перегружая её.

Решения:
1. **Single-flight / mutex** — один запрос идёт в БД, остальные ждут его результата (в Go: `golang.org/x/sync/singleflight`)
2. **Jitter к TTL** — `TTL = base + rand(0..base*0.1)` — размазываем время протухания
3. **Probabilistic Early Expiration** — за некоторое время до TTL вероятностно обновляем кэш в фоне
4. **Background refresh** — отдельный воркер обновляет кэш до протухания

</details>

---

**Чем cache-aside отличается от read-through?**

<details>
<summary>Ответ</summary>

**Cache-aside:** приложение само управляет кэшем. При miss — само идёт в БД и само пишет в кэш. Логика в коде приложения.

**Read-through:** кэш умеет сам ходить в БД при miss. Приложение работает только с кэшем, не знает про БД. Логика делегирована кэш-провайдеру.

Cache-aside популярнее — больше контроля, проще отлаживать. Read-through удобен когда кэш-провайдер поддерживает плагины загрузки (например, Ehcache, Caffeine с CacheLoader).

</details>

---

**Когда выбирать WebSocket вместо SSE?**

<details>
<summary>Ответ</summary>

**SSE** достаточен когда данные идут только от сервера к клиенту: лента новостей, уведомления, прогресс задачи.

**WebSocket** нужен когда клиент тоже активно отправляет сообщения: чат, онлайн-игры, совместное редактирование документов.

SSE проще: это обычный HTTP, работает через стандартные прокси и CDN, браузер автоматически переподключается. WebSocket требует поддержки на уровне инфраструктуры (балансировщик должен понимать WS upgrade).

</details>

---

**Что такое N+1 проблема в GraphQL?**

<details>
<summary>Ответ</summary>

Если клиент запрашивает список из N постов с авторами, GraphQL-резолвер по умолчанию делает 1 запрос за списком постов + N запросов за каждым автором.

```
GET /posts → [post1, post2, ..., postN]
GET /user/1, GET /user/2, ..., GET /user/N  ← N запросов в БД!
```

Решение — **DataLoader**: батчирует все запросы за авторами в один `SELECT * FROM users WHERE id IN (1,2,...,N)`. Реализован как отдельная библиотека практически для всех языков.

</details>
