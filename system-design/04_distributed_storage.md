# Урок 4: Распределённое хранение данных

## Репликация

Репликация — это хранение одних и тех же данных на нескольких узлах. Даёт две вещи: **отказоустойчивость** (если один узел упал — данные не потеряны) и **масштабирование чтения** (read-реплики принимают SELECT-запросы).

### Синхронная vs асинхронная

**Синхронная репликация** — master ждёт подтверждения от реплики перед тем как ответить клиенту. Данные гарантированно на двух узлах, но latency записи растёт.

**Асинхронная** — master отвечает сразу, данные на реплику доходят с задержкой. Быстро, но при падении master до репликации данные теряются (replication lag).

**Полу-синхронная (semi-sync)** — master ждёт подтверждения хотя бы от одной реплики. Компромисс между надёжностью и скоростью.

```
Sync:   Client → Master → [wait] → Replica ACK → Client OK
Async:  Client → Master → Client OK
                        ↘ Replica (позже, через replication lag)
```

### Топологии репликации

**Single-master (Master-Slave)** — один узел принимает запись, остальные только читают. Просто, но master — узкое место и единая точка отказа.

**Multi-master** — несколько узлов принимают записи. Масштабирует запись, но порождает конфликты при одновременном изменении одних данных на разных мастерах.

**Masterless (Leaderless)** — нет выделенного мастера, клиент пишет сразу на N узлов (quorum write). Cassandra, DynamoDB. Максимальная доступность, eventual consistency.

### Split Brain

Split brain — ситуация, когда два узла считают себя мастером одновременно (обычно из-за сетевого разрыва). Оба принимают записи, данные расходятся.

**Решение:** quorum — узел становится мастером только если получил подтверждение от большинства (N/2 + 1) узлов. При разрыве меньшая часть не может избрать мастера.

## Партиционирование и Шардирование

Когда данные не помещаются на один узел или один узел не справляется с нагрузкой — нужно разбить данные по узлам.

**Вертикальное партиционирование** — разные колонки в разных таблицах/базах. Например, редко используемые BLOB-поля в отдельной таблице.

**Горизонтальное партиционирование (шардирование)** — разные строки на разных узлах. Каждый шард — независимая БД с подмножеством данных.

### Стратегии шардирования

**Range-based** — диапазоны ключей на разных шардах: `id 1–1M → shard1`, `1M–2M → shard2`. Просто, хорошо для range-запросов. Проблема: hotspot (все пишут в "последний" диапазон).

**Hash-based** — `shard = hash(key) % N`. Равномерное распределение, нет hotspot. Проблема: range-запросы неэффективны, при добавлении шарда нужно перебалансировать.

**Directory-based** — отдельная таблица-маппинг `key → shard`. Максимальная гибкость, но lookup table — единая точка отказа.

### Consistent Hashing

Решает проблему перебалансировки при hash-based шардировании. Ключи и узлы размещаются на кольце. При добавлении нового узла перемещается только ~1/N ключей (а не все).

```
        0°
    K3 /   \ N1
      /     \
  N3 |  ring | K1
      \     /
    K2 \   / N2
        180°

Добавляем N4 между N1 и K1 → только ключи между N1 и N4 переезжают на N4
```

**Virtual nodes (vnodes)** — каждый физический узел занимает несколько позиций на кольце. Улучшает равномерность распределения и упрощает перебалансировку при добавлении узлов с разной мощностью.

### Роутинг запросов на шарды

- **Client-side** — клиент сам знает где шард. Быстро, нет дополнительного хопа. Сложно обновлять логику у всех клиентов.
- **Proxy** — умный прокси (Vitess, Pgpool) роутит запрос на нужный шард. Клиент не знает о шардировании.
- **Coordinator** — первый узел получает запрос, роутит на нужный шард и агрегирует ответ (Cassandra).

## CAP теорема

CAP утверждает: распределённая система может одновременно гарантировать только два из трёх свойств.

- **Consistency** — все узлы видят одни и те же данные в один момент времени
- **Availability** — каждый запрос получает ответ (не обязательно актуальный)
- **Partition Tolerance** — система работает при сетевом разрыве между узлами

> **Partition tolerance обязателен** в реальных распределённых системах — сетевые разрывы неизбежны. Поэтому реальный выбор: **CP vs AP**.

```
CP (Consistency + Partition Tolerance):
  При разрыве сети → отказываем в обслуживании, данные не расходятся
  Примеры: ZooKeeper, HBase, PostgreSQL (с синхронной репликацией)

AP (Availability + Partition Tolerance):
  При разрыве сети → отвечаем, но данные могут быть устаревшими
  Примеры: Cassandra, DynamoDB, CouchDB
```

### PACELC теорема

CAP рассматривает только поведение при сбоях. PACELC добавляет нормальный режим работы:

- **PAC** — при **P**artition: **A**vailability vs **C**onsistency (как CAP)
- **ELC** — в **E**lse (нормальный режим): **L**atency vs **C**onsistency

Пример: DynamoDB — PA/EL: при разрыве выбирает доступность, в нормальном режиме выбирает низкую latency (пишет на один узел быстро, eventual consistency).

## CDN (Content Delivery Network)

CDN — это сеть серверов, географически распределённых по миру, которые кэшируют статический контент рядом с пользователем.

```
Без CDN:   Пользователь (Москва) → сервер (США) → 150ms latency
С CDN:     Пользователь (Москва) → CDN PoP (Москва) → 5ms latency
                                        ↓ (cache miss)
                                   сервер (США) → CDN → кэш на 24ч
```

**Push CDN** — вы заранее заливаете контент на CDN-серверы. Контроль, но нужно управлять инвалидацией.

**Pull CDN** — CDN сам тянет контент с вашего origin при первом запросе, кэширует по TTL. Проще, но первый запрос медленнее (cache miss).

CDN используют не только для статики: API-ответы с правильными Cache-Control заголовками тоже кэшируются, снижая нагрузку на origin.

## CDC (Change Data Capture)

CDC — паттерн захвата изменений в БД и передачи их в другие системы. Вместо опроса БД ("что изменилось?") — подписка на поток изменений.

**Как работает:** CDC читает transaction log БД (WAL в PostgreSQL, binlog в MySQL) и публикует каждое изменение (INSERT/UPDATE/DELETE) в Kafka или другую очередь.

```
PostgreSQL WAL → [Debezium CDC] → Kafka → [Elasticsearch]  ← поиск
                                        → [Redis]           ← кэш
                                        → [Analytics DB]    ← аналитика
```

**Зачем нужен:**
- Синхронизация кэша с БД без двойной записи
- Наполнение поисковых индексов
- Аудит всех изменений
- Event sourcing

## Вопросы с собеса

**В чём разница между репликацией и шардированием?**

<details>
<summary>Ответ</summary>

**Репликация** — одни и те же данные на нескольких узлах. Цель: отказоустойчивость и масштабирование чтения. Объём данных на каждом узле одинаковый.

**Шардирование** — разные данные на разных узлах. Цель: масштабирование объёма данных и нагрузки на запись. Каждый узел хранит только часть данных.

На практике комбинируют: каждый шард реплицирован на несколько узлов.

```
Sharding + Replication:
Shard1: [Primary] + [Replica1] + [Replica2]
Shard2: [Primary] + [Replica1] + [Replica2]
```

</details>

**Объясни CAP теорему на конкретном примере.**

<details>
<summary>Ответ</summary>

Два узла БД (A и B) синхронизируются. Сеть между ними разорвалась (Partition).

**CP-система (например, ZooKeeper):** узел B не может подтвердить актуальность данных → отказывает в запросах (`503 Service Unavailable`). Зато когда сеть восстановится — данные будут консистентны.

**AP-система (например, Cassandra):** оба узла продолжают работать и принимать записи. Клиенты получают ответы, но данные на A и B могут расходиться. После восстановления сети — конфликт разрешается (last-write-wins или vector clocks).

Важно: нельзя построить "CA без P" — partition tolerance нельзя отключить, разрывы в сети неизбежны.

</details>

---

**Что такое replication lag и чем он опасен?**

<details>
<summary>Ответ</summary>

Replication lag — задержка между коммитом записи на мастере и применением этой записи на реплике. При асинхронной репликации типично несколько миллисекунд, при высокой нагрузке — секунды.

**Опасность:** клиент записал данные, получил OK, тут же прочитал с реплики — и не увидел своей записи (read-your-own-writes аномалия).

**Решения:**
- Read from master после записи (для критичных данных)
- Sticky reads — после записи некоторое время читать с мастера
- `SET SESSION TRANSACTION READ COMMITTED` на конкретный запрос с мастера

</details>

---

**Почему consistent hashing лучше обычного hash(key) % N?**

<details>
<summary>Ответ</summary>

При обычном `hash(key) % N` добавление/удаление одного узла меняет маппинг почти всех ключей:
```
5 узлов → 6 узлов: ~83% ключей переезжают на другой узел
```
Это требует массивного перемещения данных и нагружает систему.

С consistent hashing при добавлении узла переезжает только `~1/N` ключей — только те, которые "попадают" на новый узел по кольцу:
```
5 узлов → 6 узлов: ~17% ключей переезжают
```
Это особенно важно для кэшей: массовый промах (cache miss storm) при перебалансировке может обвалить БД.

</details>
