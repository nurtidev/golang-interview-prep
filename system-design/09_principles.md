# Принципы разработки: SOLID, DRY, KISS и другие

---

## Лекция

### Зачем знать эти принципы на собесе

Интервьюер задаёт вопросы про SOLID/DRY/KISS не для проверки знания аббревиатур — он проверяет, понимаешь ли ты **почему** код превращается в legacy и как этого избежать. Правильный ответ — это не определение из Википедии, а пример из практики.

---

## SOLID

Пять принципов объектно-ориентированного дизайна, сформулированных Робертом Мартином. Применимы и в Go, несмотря на отсутствие классов.

---

### S — Single Responsibility Principle (SRP)

**Принцип единственной ответственности:** у модуля/функции/структуры должна быть одна причина для изменения.

> "One reason to change" — не "делает одно дело", а "меняется по одной причине".

**Нарушение:**
```go
// UserService делает слишком много: бизнес-логику, работу с БД и отправку писем
type UserService struct{}

func (s *UserService) Register(email, password string) error {
    // 1. валидация
    // 2. хеширование пароля
    // 3. INSERT в БД
    // 4. отправка welcome-письма
    // 5. запись в аудит-лог
    return nil
}
```

**Исправление:** разделить на `UserRepository` (работа с БД), `Mailer` (письма), `AuditLogger` (логи). `UserService` только координирует.

**На собесе:** "Если появится новый способ хранения — менять `UserService`? А если поменяется шаблон письма — тоже? Это две разные причины изменений."

---

### O — Open/Closed Principle (OCP)

**Принцип открытости/закрытости:** сущности должны быть открыты для расширения, но закрыты для модификации.

Новая функциональность — через добавление кода, а не изменение существующего.

**Нарушение:**
```go
func CalculateDiscount(userType string, price float64) float64 {
    if userType == "vip" {
        return price * 0.8
    } else if userType == "premium" {
        return price * 0.9
    }
    // при добавлении нового типа — редактируем этот switch
    return price
}
```

**Исправление через интерфейс:**
```go
type DiscountPolicy interface {
    Apply(price float64) float64
}

type VIPDiscount struct{}
func (d VIPDiscount) Apply(price float64) float64 { return price * 0.8 }

type PremiumDiscount struct{}
func (d PremiumDiscount) Apply(price float64) float64 { return price * 0.9 }

// новый тип = новая структура, существующий код не трогаем
```

### L — Liskov Substitution Principle (LSP)

**Принцип подстановки Лисков:** если S — подтип T, то объекты типа T можно заменить объектами типа S без нарушения корректности программы.

Проще: **реализация интерфейса не должна нарушать его контракт**.

**Нарушение в Go:**
```go
type ReadWriter interface {
    Read() []byte
    Write(data []byte)
}

type ReadOnlyBuffer struct{}

func (b ReadOnlyBuffer) Read() []byte { return []byte("data") }
func (b ReadOnlyBuffer) Write(data []byte) {
    panic("write not supported") // нарушение LSP — клиент ожидал, что Write работает
}
```

**Правило:** если реализация не может выполнить метод интерфейса — значит интерфейс спроектирован неправильно. Нужно разбить `ReadWriter` на `Reader` и `Writer`.

---

### I — Interface Segregation Principle (ISP)

**Принцип разделения интерфейсов:** клиенты не должны зависеть от методов, которые они не используют.

В Go это особенно важно: **интерфейсы должны быть маленькими**.

**Нарушение:**
```go
type Storage interface {
    Save(data []byte) error
    Load(id string) ([]byte, error)
    Delete(id string) error
    ListAll() ([]string, error)
    Backup() error
    Restore() error
}
// тест-заглушка вынуждена реализовать все 6 методов, даже если нужен только Save
```

**Исправление:**
```go
type Saver interface {
    Save(data []byte) error
}

type Loader interface {
    Load(id string) ([]byte, error)
}
// функция принимает только то, что реально использует
func ProcessData(s Saver) { ... }
```

> В стандартной библиотеке Go это повсюду: `io.Reader`, `io.Writer`, `io.Closer` — каждый делает одно.

---

### D — Dependency Inversion Principle (DIP)

**Принцип инверсии зависимостей:** модули высокого уровня не должны зависеть от модулей низкого уровня. Оба должны зависеть от абстракций.

**Нарушение:**
```go
type OrderService struct {
    db *PostgresDB // зависимость от конкретной реализации
}
```

**Исправление:**
```go
type OrderRepository interface {
    Save(order Order) error
    FindByID(id string) (Order, error)
}

type OrderService struct {
    repo OrderRepository // зависимость от абстракции
}
// в тестах подставляем MockRepository, в проде — PostgresRepository
```

## DRY — Don't Repeat Yourself

**Не повторяй себя:** каждый кусок знания должен иметь единственное, однозначное представление в системе.

DRY — про **знания**, не про код. Два похожих цикла — это не нарушение DRY. Нарушение — когда одно бизнес-правило живёт в двух местах.

**Нарушение:**
```go
// логика валидации email дублируется в регистрации и обновлении профиля
func Register(email string) error {
    if !strings.Contains(email, "@") { return errors.New("invalid email") }
    ...
}

func UpdateProfile(email string) error {
    if !strings.Contains(email, "@") { return errors.New("invalid email") }  // копипаста
    ...
}
```

**Исправление:** вынести `validateEmail()` в одно место.

**Антипаттерн — избыточное применение DRY:** иногда дублирование лучше неправильной абстракции. Если два куска кода похожи, но меняются по разным причинам — объединять их преждевременно.

## KISS — Keep It Simple, Stupid

**Держи это простым:** большинство систем работают лучше, если они простые, а не сложные. Простота должна быть целью дизайна.

```go
// ПЛОХО: преждевременная абстракция
type StrategyFactory interface {
    CreateStrategy(ctx context.Context, opts ...Option) (Strategy, error)
}

// ХОРОШО: если у нас одна реализация — просто функция
func processOrder(order Order) error {
    // прямолинейный код
}
```

**На собесе:** KISS нарушается когда пишут "на вырост" — добавляют слои абстракции, фабрики, конфигурации для кода, который никогда не менялся и не будет меняться.

---

## YAGNI — You Aren't Gonna Need It

**Тебе это не понадобится:** не добавляй функциональность, пока она действительно не нужна.

Связан с KISS, но фокус другой: KISS про сложность существующего кода, YAGNI — про запрет добавлять то, что "может пригодиться".

**Нарушение:**
```go
// "вдруг захотим поменять реализацию логгера"
type Logger interface { Log(msg string) }
type ConsoleLogger struct{}
type FileLogger struct{}     // пока не используется
type RemoteLogger struct{}   // пока не используется
```

Если реальной потребности нет — просто `log.Println`. Когда потребность появится — рефакторинг займёт 20 минут.

## TDD — Test-Driven Development

**Разработка через тестирование** — методология, где тесты пишутся **до** кода.

### Цикл Red → Green → Refactor

```
1. RED    — написать тест, который падает (функция ещё не существует)
2. GREEN  — написать минимальный код, чтобы тест прошёл
3. REFACTOR — улучшить код, не ломая тесты
```

```go
// 1. RED: пишем тест
func TestAdd(t *testing.T) {
    result := Add(2, 3)
    assert.Equal(t, 5, result) // компилятор ругается: Add не существует
}

// 2. GREEN: минимальная реализация
func Add(a, b int) int {
    return a + b
}

// 3. REFACTOR: если нужно — улучшаем
```

**Плюсы TDD:**
- Код изначально тестируемый (нет "как теперь это протестировать?")
- Тесты документируют поведение
- Меньше страха рефакторинга

**Минусы:**
- Медленнее в начале
- Сложно применять к I/O-операциям без моков
- Требует дисциплины

**BDD (Behavior-Driven Development)** — расширение TDD, где тесты описывают поведение с точки зрения пользователя: `Given / When / Then`.

## Cohesion & Coupling

Два фундаментальных понятия, к которым сводятся все принципы выше.

**Coupling (связанность)** — насколько сильно модули зависят друг от друга.
- Высокий coupling: изменение одного модуля ломает другой.
- Цель: **низкий coupling**.

**Cohesion (связность)** — насколько элементы внутри модуля относятся к одной задаче.
- Низкая cohesion: структура `UserOrderPaymentEmailLogger` делает всё подряд.
- Цель: **высокая cohesion**.

> **Правило:** High Cohesion + Low Coupling = хорошо спроектированная система.

## Шпаргалка

| Принцип | Одна фраза | Нарушение выглядит как |
|---------|-----------|----------------------|
| **SRP** | Одна причина для изменений | `UserService` на 500 строк |
| **OCP** | Расширяй, не меняй | `switch userType` с 10 case |
| **LSP** | Реализация соблюдает контракт | `panic("not implemented")` |
| **ISP** | Маленькие интерфейсы | интерфейс на 15 методов |
| **DIP** | Зависи от абстракции | `struct { db *PostgresDB }` |
| **DRY** | Одно знание — одно место | одна логика в 3 местах |
| **KISS** | Простота лучше сложности | 5 слоёв абстракции для CRUD |
| **YAGNI** | Не пиши на вырост | интерфейсы без второй реализации |
| **TDD** | Тест → код → рефакторинг | "напишу тесты потом" |

## Вопросы с собесов

**Q: Объясни SOLID простыми словами.**
A: Пять принципов, которые делают код расширяемым без переписывания. SRP — каждый модуль делает одно дело. OCP — новое поведение добавляем кодом, не правим старый. LSP — реализации не нарушают контракт интерфейса. ISP — интерфейсы маленькие. DIP — зависим от абстракций, а не конкретики.

**Q: Чем DRY отличается от KISS?**
A: DRY про дублирование знаний — одно бизнес-правило в одном месте. KISS про сложность — не добавляй лишних абстракций. Можно соблюдать DRY, нарушая KISS: вынести повторяющийся код в абстрактную фабрику с 10 параметрами.

**Q: Когда нарушение DRY оправдано?**
A: Когда два похожих куска кода меняются по разным причинам. Преждевременное обобщение создаёт связность там, где её не должно быть. "Duplicaton is far cheaper than the wrong abstraction" — Sandi Metz.

**Q: В чём смысл TDD, если можно написать тесты после?**
A: TDD вынуждает думать об интерфейсе до реализации — код получается тестируемым по природе. Тесты после часто пишутся "под существующий код", а не проверяют поведение. Плюс цикл red-green даёт мгновенную обратную связь.

**Q: Как ISP проявляется в стандартной библиотеке Go?**
A: `io.Reader` — один метод `Read`. `io.Writer` — один метод `Write`. `io.Closer` — один метод `Close`. Их комбинируют: `io.ReadWriter`, `io.ReadWriteCloser`. Это эталон ISP.
