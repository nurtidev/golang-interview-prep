# Урок 5: Паттерны и приёмы проектирования

## Архитектура бэкенда: Монолит vs Микросервисы

**Монолит** — всё приложение — один деплоящийся артефакт. Просто разрабатывать и дебажить, нет сетевых вызовов между компонентами. Проблемы начинаются с ростом команды и кодовой базы: долгие сборки, страх изменений, невозможность масштабировать только один компонент.

**Микросервисная архитектура (MSA)** — система из небольших независимых сервисов, каждый со своей БД и своим деплоем. Независимое масштабирование, изоляция отказов, можно выбрать разный стек для каждого сервиса. Цена: операционная сложность, сетевые вызовы, распределённые транзакции, необходимость service discovery.

> Правило: начинай с монолита, переходи на микросервисы когда монолит становится проблемой — не раньше.

| | Монолит | Микросервисы |
|---|---|---|
| Деплой | один артефакт | независимо на сервис |
| Масштабирование | всё или ничего | точечно |
| Транзакции | простые (ACID) | сложные (SAGA, 2PC) |
| Latency | нет сетевых вызовов | +N сетевых хопов |
| Команда | малая | каждый сервис — своя команда |

## Event-Driven Architecture

Событийно-ориентированная архитектура — сервисы общаются через события, а не прямые вызовы. Это развязывает зависимости: издатель не знает о подписчиках.

**Event Notification** — "что-то произошло". Событие содержит минимум данных: `OrderCreated { orderId: 42 }`. Подписчик идёт за деталями сам. Минимальная связность, но дополнительный запрос.

**Event-Carried State Transfer** — событие содержит все нужные данные: `OrderCreated { orderId: 42, userId: 7, items: [...] }`. Подписчик не делает дополнительных запросов, но события тяжелее.

**Event Collaboration** — сервисы совместно реализуют бизнес-процесс через цепочку событий. OrderService → `OrderCreated` → PaymentService → `PaymentCompleted` → ShippingService.

## SAGA — распределённые транзакции

В микросервисах нет единой БД, поэтому ACID-транзакции через несколько сервисов невозможны. SAGA — это последовательность локальных транзакций, каждая с компенсирующей операцией (rollback).

**Хореография (Choreography)** — сервисы сами реагируют на события, нет центрального координатора:

```
OrderService  →  order.created  →  PaymentService
                                        ↓
                               payment.completed  →  InventoryService
                                                          ↓
                               payment.failed    →  OrderService (компенсация: отменить заказ)
```

Плюс: нет единой точки отказа. Минус: сложно отследить весь flow, легко получить "спагетти" из событий.

**Оркестрация (Orchestration)** — центральный оркестратор управляет всем процессом:

```
                    ┌─────────────────┐
                    │   OrderSaga     │
                    │  (оркестратор)  │
                    └─────────────────┘
                     /        |        \
              Payment    Inventory    Shipping
```

Плюс: весь flow виден в одном месте, легко управлять компенсациями. Минус: оркестратор — центральная точка отказа и знает много о других сервисах.

## 2PC (Two-Phase Commit)

2PC — протокол, гарантирующий что все участники либо коммитят, либо откатывают транзакцию. Координатор управляет двумя фазами:

```
Фаза 1 (Prepare):
  Coordinator → "готов закоммитить?" → Service A, Service B, Service C
  Все отвечают "YES" или "NO"

Фаза 2 (Commit/Abort):
  Все ответили YES → Coordinator → "COMMIT"
  Хоть один NO    → Coordinator → "ABORT"
```

**Проблема 2PC:** если координатор падает после фазы 1 — участники заблокированы в неопределённом состоянии до его восстановления. Это делает 2PC непригодным для высоконагруженных систем. На практике предпочитают SAGA.

## Distributed Locking и Leader Election

**Распределённая блокировка** нужна, когда несколько инстансов сервиса не должны выполнять одно действие одновременно (например, крон-задача, которая запускается на 10 серверах).

Реализация через Redis (`SET key value NX PX 30000` — атомарный SET if Not eXists с TTL):
```
Инстанс 1: SET lock:cron "inst1" NX PX 30000 → OK    ← получил блокировку
Инстанс 2: SET lock:cron "inst2" NX PX 30000 → NULL  ← занято, пропускает
```

TTL критичен: если инстанс умер держа блокировку — TTL освободит её автоматически.

**Leader Election** — частный случай distributed lock: из N инстансов один становится "лидером" и выполняет особую работу. Остальные в режиме standby. Реализуется через ZooKeeper, etcd или тот же Redis.

## Паттерны отказоустойчивости

### Circuit Breaker

Circuit breaker — "автоматический выключатель" для внешних вызовов. Если сервис начал отказывать — перестаём его дёргать, отвечаем fallback-ом, даём ему время восстановиться.

```
CLOSED (нормальная работа)
  → слишком много ошибок → OPEN (отклоняем запросы, не идём в сервис)
  → после таймаута → HALF-OPEN (пробуем один запрос)
  → успех → CLOSED
  → ошибка → OPEN снова
```

Без circuit breaker: медленный зависший сервис держит наши потоки, ресурсы исчерпываются — каскадный отказ всей системы.

### Rate Limiting

Rate limiting — ограничение количества запросов от одного клиента/IP за единицу времени. Защита от DDoS, злоупотреблений, перегрузки.

**Алгоритмы:**
- **Fixed Window** — счётчик сбрасывается каждую минуту. Проблема: burst на границе окна (50 запросов в конце минуты + 50 в начале следующей = 100 за секунду).
- **Sliding Window** — скользящее окно, нет проблемы с границами. Дороже по памяти.
- **Token Bucket** — "ведро" с токенами, которые пополняются с постоянной скоростью. Разрешает короткие burst-ы, популярен (AWS API Gateway, Nginx).
- **Leaky Bucket** — запросы обрабатываются с постоянной скоростью, излишки отбрасываются. Сглаживает трафик.

### Retry с Backoff

Не все ошибки постоянные: сеть моргнула, сервис перезагружается. Retry даёт второй шанс.

**Exponential Backoff с Jitter:**
```
Попытка 1: сразу
Попытка 2: wait 1s  + rand(0..0.5s)
Попытка 3: wait 2s  + rand(0..1s)
Попытка 4: wait 4s  + rand(0..2s)
Попытка 5: wait 8s  + rand(0..4s)  → сдаёмся
```

Jitter (случайный разброс) обязателен: без него N инстансов после сбоя одновременно стартуют retry и снова перегружают восстанавливающийся сервис.

**Важно:** retry только для идемпотентных операций. GET — retry безопасен. POST (создание) — без idempotency key retry создаст дубликаты.

### Graceful Degradation

Система продолжает работать при частичном отказе, отдавая деградированный (но полезный) результат вместо полного отказа.

```
Лента новостей:
  Персонализация недоступна → показываем общую ленту (не персональную)
  Картинки не загружаются  → показываем текст без картинок
  Полный отказ БД          → показываем кэшированную версию из 10 минут назад
```

## Паттерны деплоя

**Rolling Deploy** — постепенно заменяем инстансы старой версии новой. В любой момент часть серверов на старой версии, часть — на новой. Нет downtime, но backward compatibility обязательна.

**Blue-Green Deploy** — два идентичных окружения: blue (текущий продакшн) и green (новая версия). Переключаем трафик разом с blue на green. Instant rollback — просто вернуть трафик на blue. Дорого: нужно двойное количество ресурсов.

**Canary Deploy** — новая версия получает малую долю трафика (1–5%). Мониторим метрики, постепенно увеличиваем до 100%. Позволяет поймать проблему до того, как она затронет всех пользователей.

```
Rolling:      [v1][v1][v2][v2][v2]  → [v2][v2][v2][v2][v2]
Blue-Green:   blue=100%             → green=100% (мгновенно)
Canary:       [v1 95%] + [v2 5%]   → [v1 50%] + [v2 50%] → [v2 100%]
```

## Вопросы с собеса

**Когда использовать SAGA вместо 2PC?**

<details>
<summary>Ответ</summary>

**2PC** — синхронный протокол. Все участники блокированы до завершения транзакции. При падении координатора — участники зависают. Хорош только когда все сервисы поддерживают XA-транзакции и нагрузка невысокая.

**SAGA** — асинхронный, через события или оркестратор. Нет глобальной блокировки. Eventual consistency вместо strong consistency. При ошибке — компенсирующие транзакции.

Выбираем SAGA в микросервисах почти всегда: 2PC — антипаттерн для высоконагруженных распределённых систем из-за блокировок и сложности с отказами координатора.

</details>

---

**Объясни Circuit Breaker: зачем нужен и как работает?**

<details>
<summary>Ответ</summary>

Circuit breaker защищает от каскадных отказов. Без него: медленный сервис B → наши запросы к нему висят → пул потоков/соединений исчерпывается → сервис A тоже перестаёт отвечать → каскад.

Три состояния:
- **CLOSED** — нормальная работа, запросы проходят, ведётся подсчёт ошибок
- **OPEN** — слишком много ошибок (например, >50% за 10 секунд), запросы отклоняются немедленно (fallback), сервис B не дёргается
- **HALF-OPEN** — после таймаута (например, 30 сек) пропускаем один пробный запрос. Успех → CLOSED, ошибка → OPEN

Реализации: Hystrix (Java), go-hystrix, resilience4j.

</details>

---

**Чем Canary отличается от Blue-Green деплоя?**

<details>
<summary>Ответ</summary>

**Blue-Green** — переключение трафика происходит мгновенно и полностью. Хорош для: простого rollback, когда нужно одномоментно обновить все инстансы. Минус: нельзя постепенно проверить новую версию под реальной нагрузкой.

**Canary** — постепенное увеличение трафика на новую версию. Позволяет поймать проблемы на 1–5% пользователей, не затрагивая всех. Минус: сложнее инфраструктурно, нужен умный балансировщик. Дольше деплой.

На практике: Blue-Green для простых изменений, Canary для рискованных или крупных изменений.

</details>

**Как реализовать rate limiting на уровне сервиса?**

<details>
<summary>Ответ</summary>

**Локально** (в памяти процесса) — быстро, но при N инстансах лимит умножается на N: 100 rps/user * 10 инстансов = 1000 rps реально.

**Централизованно** (Redis) — точный подсчёт, работает для всех инстансов. Небольшой сетевой overhead (~1 мс).

Алгоритм Token Bucket на Redis:
```
EVAL script:
  tokens = GET key
  if tokens > 0: DECR key, return ALLOW
  else: return DENY
SET key N EX window  -- инициализация при первом запросе
```

В продакшне: обычно используют готовые решения — nginx `limit_req_zone`, API Gateway (Kong, AWS API GW), или библиотеки типа `golang.org/x/time/rate` для локального.

</details>
