# System Design: основы и типовые задачи

---

## Лекция

### 1. Фреймворк решения задачи (4 шага)

На интервью нельзя сразу рисовать схему. Сначала — уточнения.

**Шаг 1 — Уточнение требований (5 минут)**
```
- Сколько пользователей? DAU/MAU?
- Какой RPS на чтение и запись?
- Какая допустимая задержка (p99)?
- Нужна ли глобальная доступность?
- Допустима ли потеря данных? (durability)
- Нужна ли строгая консистентность или eventual consistency?
```

**Шаг 2 — Оценка масштаба (back of napkin)**
```
Пример: 10M DAU, каждый делает 10 запросов/день
→ RPS = 10M * 10 / 86400 ≈ 1200 RPS средний
→ Пиковый RPS = средний * 3-5 ≈ 5000-6000 RPS

Хранилище: 10M пользователей * 1KB профиль = 10GB → легко в одной БД
Но: 10M заказов/день * 365 * 5 лет * 1KB = ~18TB → нужно шардирование
```

**Шаг 3 — Высокоуровневая архитектура**
- API Gateway / Load Balancer
- Сервисы (stateless, горизонтально масштабируемые)
- Хранилище (выбор БД под нагрузку)
- Кэш (Redis)
- Очереди (Kafka)

**Шаг 4 — Детали и узкие места**
- Где bottleneck?
- Как шардировать?
- Что будет при сбое одного компонента?
- Как мониторить?

### 2. CAP теорема

Распределённая система может гарантировать только два из трёх:

| Свойство | Описание |
|----------|----------|
| **C**onsistency | Все узлы видят одни и те же данные одновременно |
| **A**vailability | Система отвечает на каждый запрос (не обязательно актуальными данными) |
| **P**artition tolerance | Система работает при потере части сообщений между узлами |

В реальности сеть всегда может разделиться (P неизбежно), поэтому выбор между **CP** и **AP**:

- **CP (Consistency + Partition):** PostgreSQL, ZooKeeper, HBase. При разделении сети — отказываем в обслуживании, но данные консистентны.
- **AP (Availability + Partition):** Cassandra, DynamoDB, CouchDB. При разделении — отвечаем (возможно устаревшими данными), потом синхронизируемся.

**Для e-commerce (Ozon):**
- Каталог товаров → AP (можно показать устаревшую цену, потом обновим)
- Баланс кошелька → CP (деньги должны быть консистентны)
- Корзина → AP (eventual consistency допустима)

### 3. Eventual Consistency

После записи данные не сразу видны всем узлам — но **в конечном счёте** станут консистентны.

```
Запись в мастер → репликация на реплики → (задержка) → реплики обновились

Пользователь читает с реплики сразу после записи → видит старые данные
Пользователь читает через 100ms → видит новые данные
```

**Где приемлемо:** лента новостей, рекомендации, счётчики просмотров, каталог.
**Где неприемлемо:** платежи, остатки товаров, инвентаризация.

### 4. Типовые задачи на интервью Ozon

---

#### Задача А: Система уведомлений для 10M пользователей

**Требования:**
- Push, Email, SMS уведомления
- Шаблоны уведомлений
- Приоритеты (транзакционные vs маркетинговые)
- Retry при неудаче

**Архитектура:**

```
[API Service] → [Notification Service] → [Kafka]
                                              ↓
                              ┌───────────────┼──────────────┐
                         [Push Worker]  [Email Worker]  [SMS Worker]
                              ↓               ↓              ↓
                         [Firebase]       [SendGrid]     [Twilio]
```

**Ключевые решения:**
- Kafka как буфер — сглаживает пики нагрузки (Black Friday)
- Разные топики для разных приоритетов (транзакционные не ждут в очереди за маркетингом)
- Retry с exponential backoff через Dead Letter Queue
- Дедупликация: хранить sent_notification_id чтобы не слать дважды
- Rate limiting на уровне пользователя (не более 10 push в час)

**БД:**
```sql
notifications (id, user_id, type, template_id, status, created_at, sent_at)
templates (id, name, subject, body, channel)
user_preferences (user_id, channel, enabled)
```

---

#### Задача Б: Корзина покупок во время Black Friday

**Требования:**
- 100K RPS на чтение корзины
- 10K RPS на изменение
- Корзина сохраняется между сессиями
- При добавлении товара — проверка остатка

**Архитектура:**

```
[Client] → [API Gateway] → [Cart Service]
                                 ↓
                    ┌────────────┼────────────┐
                 [Redis]    [PostgreSQL]   [Stock Service]
               (горячие      (персистентное   (проверка
               корзины)       хранилище)       остатка)
```

**Ключевые решения:**

```
Read path:  Redis (TTL 7 дней) → если miss → PostgreSQL → пишем в Redis
Write path: Пишем в Redis + асинхронно в PostgreSQL через очередь
```

- Redis Hash для корзины: `cart:{user_id}` → `{product_id: quantity}`
- Stock Service: проверяем остаток при добавлении, но не резервируем — резервируем только при оформлении заказа
- Проблема overselling: используем Redis DECR атомарно при оформлении

**Что если Redis упал:**
- Fallback на PostgreSQL (медленнее, но данные не теряем)
- Circuit Breaker вокруг Redis

---

#### Задача В: Система ранжирования товаров (поиск)

**Требования:**
- 50K RPS поисковых запросов
- Результаты за < 200ms
- Фильтры: категория, цена, рейтинг, наличие

**Архитектура:**

```
[Client] → [Search Service] → [Elasticsearch]
                ↓
          [Ranking Service] → [ML Model]
                ↓
          [Cache (Redis)]   ← популярные запросы
```

**Ключевые решения:**
- Elasticsearch для полнотекстового поиска и фильтрации
- Отдельный сервис ранжирования — ML модель учитывает CTR, рейтинг, маржу
- Кэш популярных запросов (топ-10% запросов = 80% трафика)
- Индексация товаров асинхронно через Kafka (изменение цены → событие → переиндексация)

### 5. Выбор базы данных

| Сценарий | База данных | Почему |
|----------|-------------|--------|
| Транзакции, заказы, финансы | **PostgreSQL** | ACID, сложные запросы, JOIN |
| Сессии, корзины, кэш | **Redis** | Скорость, TTL, атомарные операции |
| Логи, события, временные ряды | **ClickHouse** | Колоночное хранение, быстрая аналитика |
| Каталог с поиском | **Elasticsearch** | Full-text, фильтры, ранжирование |
| Чаты, социальный граф | **Cassandra** | Высокая запись, wide column |
| Документы (профили) | **MongoDB** | Гибкая схема, вложенные данные |
| Очереди сообщений | **Kafka** | Надёжность, replay, партиционирование |

### 6. Паттерны масштабирования

**Горизонтальное масштабирование (scale out):**
- Добавляем копии stateless сервисов за Load Balancer
- БД: read replicas для чтения

**Шардирование:**
```
По user_id: user_id % N → shard_N
По диапазону: user_id 0-1M → shard_1, 1M-2M → shard_2
По хэшу:     hash(user_id) % N → равномерное распределение
```

**Проблема горячих ключей:** один shard перегружен (популярный продавец, вирусный товар). Решение: добавить суффикс к ключу (`product_id:1`, `product_id:2`) и агрегировать при чтении.

**Caching стратегии:**

```
Cache-Aside:  читаем кэш → miss → читаем БД → пишем в кэш
Write-Through: пишем в БД и кэш синхронно
Write-Behind:  пишем в кэш → асинхронно в БД (риск потери данных)
```

**Transactional Outbox (реальный вопрос из Ozon):**
Проблема: как атомарно сохранить запись в БД и отправить событие в Kafka?

```sql
-- В одной транзакции:
BEGIN;
INSERT INTO orders (id, status) VALUES (1, 'created');
INSERT INTO outbox (event_type, payload) VALUES ('order.created', '{"id":1}');
COMMIT;

-- Отдельный воркер читает outbox и публикует в Kafka
-- При успехе — помечает запись как published
```

## Вопросы с интервью

**Q: Как спроектировать систему для Black Friday с нагрузкой в 10x от обычной?**
A: Заранее нагрузочное тестирование, горизонтальное масштабирование сервисов через HPA в Kubernetes, кэширование популярных данных в Redis, очереди для сглаживания пиков (Kafka), graceful degradation — при перегрузке показывать упрощённую версию (без рекомендаций, без персонализации).

**Q: Как обеспечить exactly-once доставку сообщений в Kafka?**
A: Transactional producer + idempotent consumer. Producer пишет с транзакцией, consumer при обработке проверяет idempotency key в Redis/БД. Если ключ уже есть — пропускаем, считаем обработанным.

**Q: Чем отличается партиционирование от шардирования?**
A: Партиционирование — внутри одного сервера, управляется Postgres, прозрачно для приложения. Шардирование — данные на разных серверах, маршрутизацию делает приложение или прокси (Vitess, Citus). Партиционирование не масштабирует запись, только улучшает производительность запросов.

**Q: Что выбрать — микросервисы или монолит?**
A: Начинать с монолита. Переходить на микросервисы когда: разные части системы требуют разного масштабирования, разные команды независимо деплоят, нужна технологическая гетерогенность. Микросервисы добавляют сложность (сеть, трассировка, distributed transactions) — оправданы только при реальной необходимости.

**Q: Как обработать ситуацию когда платёж прошёл, но заказ не создался?**
A: Паттерн Saga. Либо choreography (сервисы обмениваются событиями и делают компенсирующие транзакции при сбое), либо orchestration (центральный оркестратор управляет шагами). При сбое создания заказа — отправить компенсирующее событие на возврат платежа.
