# Урок 6: Проектирование ленты и мессенджера

## Лента друзей (ВКонтакте / Instagram)

### Требования

**Функциональные:**
- Пользователь публикует пост
- Пользователь видит ленту из постов своих друзей, отсортированных по времени
- Лайки, комментарии

**Нефункциональные:**
- 50M DAU, у каждого пользователя в среднем 300 друзей
- Лента загружается < 200 мс
- Посты доступны сразу после публикации (eventual consistency — нормально)

### Главная проблема: Fan-Out

Когда пользователь публикует пост — его нужно "доставить" в ленты всех его друзей. При 300 друзьях — 300 операций. Но у знаменитостей миллионы подписчиков: одна публикация → миллион записей. Это называется **Fan-Out on Write**.

Есть два подхода, и у каждого свои проблемы:

**Fan-Out on Write (Push)** — при публикации сразу записываем пост в кэш ленты каждого подписчика:
```
Post published → для каждого подписчика → добавить post_id в Redis List[user_id]

Чтение ленты: GET Redis List[user_id] → быстро, O(1)
Запись: O(N подписчиков) — медленно для знаменитостей
```

**Fan-Out on Read (Pull)** — при открытии ленты идём к каждому другу и собираем его последние посты:
```
Read feed → для каждого друга → SELECT posts WHERE user_id=friend ORDER BY time LIMIT 10
          → merge N списков → ленты

Чтение: O(N друзей) — медленно
Запись: O(1) — мгновенно
```

### Гибридный подход (используют Instagram, Twitter)

Обычных пользователей — push (их посты идут в кэш подписчиков). Знаменитостей с >10K подписчиков — pull (при чтении ленты подмешиваем их посты отдельно):

```
Публикация обычного пользователя:
  post → Kafka → FanOut Worker → Redis List для каждого подписчика

Публикация знаменитости:
  post → DB (только запись поста)

Чтение ленты пользователя:
  1. Redis List[user_id]           ← посты обычных друзей (push)
  2. SELECT от знаменитостей        ← pull для celebrities
  3. Merge + сортировка по времени
  4. Вернуть первые 20
```

### Хранение данных

```
posts:     { post_id, user_id, content, media_url, created_at }
           PostgreSQL + партиционирование по дате

feed_cache: Redis Sorted Set { key: feed:{user_id}, score: timestamp, member: post_id }
            Храним только последние 1000 постов, старше — из БД

media:      S3 / object storage, CDN для раздачи
```

### Архитектура

```
Client → API Gateway → Feed Service
                           ├── Redis (горячая лента)
                           ├── Post DB (PostgreSQL)
                           └── Kafka → FanOut Worker → Redis (обновление лент)

Публикация: Post Service → Kafka → [FanOut Worker, Notification Service, Search Indexer]
```

## WhatsApp (мессенджер)

### Требования

**Функциональные:**
- Отправка сообщений 1-на-1 и групповые чаты
- Online/offline статус, "прочитано"
- Push-уведомления при офлайн

**Нефункциональные:**
- 2B пользователей, 100B сообщений в день (~1.2M msg/сек)
- Доставка < 500 мс
- Сообщения не теряются (at-least-once)
- E2E шифрование

### Хранение сообщений

Каждое сообщение хранится у **получателя** (не у отправителя). Для группового чата — копия у каждого участника. Это позволяет быстро читать историю чата одного пользователя без JOIN-ов.

```
messages: { message_id, chat_id, sender_id, content_encrypted, created_at, status }
          Cassandra: ключ партиции = chat_id, кластеризация по created_at DESC
          → все сообщения одного чата на одном узле, чтение по времени эффективно
```

### Доставка сообщений

Клиент держит постоянное WebSocket-соединение с сервером. При отправке сообщения:

```
Sender → WebSocket → Chat Server A
                         ├── сохранить в Cassandra
                         ├── если получатель онлайн на Server B → Message Queue → Server B → WebSocket → Receiver
                         └── если офлайн → Push Notification (APNs/FCM)

ACK flow:
  Сервер получил    → ✓  (один tick)
  Получатель получил → ✓✓ (два tick)
  Получатель прочитал → ✓✓ blue
```

### Статус онлайн/офлайн

Каждый клиент каждые 5 секунд отправляет heartbeat. Сервер обновляет `last_seen` в Redis:
```
Redis: SET user:online:{user_id} 1 EX 10   ← истекает через 10 сек без heartbeat
```

Но при 2B пользователях показывать статус всем — слишком дорого. WhatsApp показывает статус только "взаимным" контактам и только когда открыт чат.

### Групповые чаты

Для небольших групп (< 500 участников) — Fan-Out на сервере: сообщение копируется каждому участнику. Для больших групп (трансляции) — подписочная модель: участники сами тянут сообщения.

### Архитектура

```
Client ──WebSocket──► Chat Server (stateful — хранит соединения)
                           │
                    ┌──────┴──────┐
                Message Queue   Cassandra
                (Kafka/Redis)    (сообщения)
                    │
               Push Service ──► APNs / FCM
```

## Вопросы с собеса

**Почему для ленты выбирают гибридный Fan-Out, а не чистый push или pull?**

<details>
<summary>Ответ</summary>

**Чистый Push (Fan-Out on Write):** одна публикация Дженнифер Лопес (100M подписчиков) → 100M записей в Redis. За несколько секунд. Система не успевает, лента у подписчиков обновляется с задержкой в минуты.

**Чистый Pull (Fan-Out on Read):** пользователь с 1000 друзей открывает ленту → 1000 запросов в БД → merge → задержка 500+ мс. Кэшировать сложно — у каждого пользователя уникальная лента.

**Гибрид:** разделяем по threshold (например, 10K подписчиков). Обычные пользователи — push (быстро, их мало). Знаменитости — pull при чтении (дополнительный запрос, но он один, не миллион). Лучший баланс.

</details>

---

**Почему для сообщений чатов выбирают Cassandra, а не PostgreSQL?**

<details>
<summary>Ответ</summary>

Паттерн доступа к сообщениям: всегда читаем/пишем по `chat_id` + сортировка по времени. Cassandra оптимизирована именно под это:

- Ключ партиции = `chat_id` → все сообщения чата на одном узле, нет распределённых запросов
- Clustering key = `created_at DESC` → данные физически отсортированы, чтение последних N сообщений — O(N)
- Write-optimized: LSM-дерево вместо BTree, запись всегда быстрая

PostgreSQL: JOIN-ы не нужны, ACID-транзакции не нужны (потеря одного сообщения обрабатывается на уровне приложения через retry), горизонтальное масштабирование сложнее.

При 1.2M msg/сек PostgreSQL потребует серьёзного шардирования руками, Cassandra масштабируется горизонтально нативно.

</details>

**Как обеспечить доставку сообщения если получатель офлайн?**

<details>
<summary>Ответ</summary>

1. Сообщение сохраняется в Cassandra со статусом `SENT`
2. Сервер видит что получатель офлайн (нет WebSocket-соединения) → отправляет Push Notification через APNs (iOS) или FCM (Android)
3. Пользователь открывает приложение → устанавливает WebSocket → сервер отдаёт все сообщения со статусом `DELIVERED = false` → статус обновляется

Гарантия at-least-once: если push-уведомление не дошло — при следующем открытии приложения клиент сам запрашивает новые сообщения. Сообщение в Cassandra не удаляется до явного ACK от клиента.

</details>

**Как масштабировать stateful Chat Servers (хранят WebSocket-соединения)?**

<details>
<summary>Ответ</summary>

WebSocket — постоянное соединение к конкретному серверу. Если пользователь A подключён к Server1, а пользователь B к Server2 — как Server1 доставит сообщение на Server2?

**Решение — межсерверный Message Bus (Kafka или Redis Pub/Sub):**
```
Server1 (A подключён) → Redis Pub/Sub channel:"user:B" → Server2 (B подключён) → WebSocket → B
```

**Service Discovery:** отображение `user_id → server_id` в Redis. При получении сообщения для пользователя B — ищем на каком сервере он сидит, публикуем туда.

Балансировщик для WebSocket должен поддерживать sticky sessions (или умеет делать upgrade по IP-хешу).

</details>
