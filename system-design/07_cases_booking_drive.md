# Урок 7: Проектирование Booking.com и Google Drive

## Booking.com (система бронирования)

### Требования

**Функциональные:**
- Поиск отелей по городу, датам, фильтрам
- Просмотр доступности и цен
- Бронирование с оплатой
- Отмена брони

**Нефункциональные:**
- 500M посещений/месяц, пиковая нагрузка — праздники
- Поиск < 300 мс
- **Нельзя допустить двойное бронирование** (один номер дважды на одни даты)
- Eventual consistency для поиска — нормально, строгая консистентность для бронирования — обязательна

### Главная проблема: предотвращение двойного бронирования

Два пользователя одновременно бронируют последний номер. Без защиты оба получат подтверждение — это катастрофа.

**Вариант 1: Пессимистичная блокировка**
```sql
BEGIN;
SELECT * FROM inventory
WHERE hotel_id=1 AND room_type='standard' AND date='2024-12-25'
FOR UPDATE;  ← блокируем строку

-- проверяем что rooms_available > 0
UPDATE inventory SET rooms_available = rooms_available - 1 ...;
COMMIT;
```
Надёжно, но блокировка снижает параллелизм. Подходит для высококонкурентных номеров.

**Вариант 2: Оптимистичная блокировка (версионирование)**
```sql
-- Читаем с version
SELECT rooms_available, version FROM inventory WHERE ... → {available: 1, version: 42}

-- Обновляем только если version не изменилась
UPDATE inventory SET rooms_available = 0, version = 43
WHERE hotel_id=1 AND date='2024-12-25' AND version = 42;
-- affected rows = 0 → кто-то успел раньше → retry или ошибка
```
Нет блокировок, высокий параллелизм, но нужны retry при конфликтах.

**Вариант 3: Idempotent reservation через уникальный ключ**
```sql
INSERT INTO reservations (reservation_id, hotel_id, room_id, check_in, check_out, user_id)
VALUES (...)
-- UNIQUE constraint на (room_id, check_in, check_out) не позволит вставить дубль
```

### Хранение доступности

Таблица `inventory` — не "номера", а "слоты доступности":
```
inventory:
  hotel_id | room_type | date       | total_rooms | reserved
  100      | standard  | 2024-12-25 | 50          | 49

-- Атомарное бронирование:
UPDATE inventory SET reserved = reserved + 1
WHERE hotel_id=100 AND room_type='standard' AND date='2024-12-25'
AND reserved < total_rooms
```

### Поиск отелей

Поиск по городу + датам + фильтрам — это OLAP-запрос с геопоиском. PostgreSQL плохо справляется с полнотекстовым поиском по 500k отелей. Решение — **Elasticsearch**:

```
Отель обновился → CDC (Debezium) → Kafka → ES Indexer → Elasticsearch

Поиск:
  ES: "отели в Москве, с бассейном, от 3000₽, доступны 25-27 декабря"
  → список hotel_id
  → PostgreSQL: GET prices/details для этих hotel_id
  → результат пользователю
```

### Процесс бронирования (SAGA)

```
1. OrderService: создать бронь со статусом PENDING
2. InventoryService: заблокировать номер (reserved + 1)
3. PaymentService: провести оплату
   → успех: OrderService обновить статус → CONFIRMED
   → провал: InventoryService компенсация (reserved - 1), статус → FAILED
```

```
BookingService → Kafka
    ├── InventoryService: lock_room
    │       → success: emit room_locked
    │       → fail: emit room_lock_failed → компенсация
    ├── PaymentService: charge
    │       → success: emit payment_done
    │       → fail: emit payment_failed → разблокировать номер
    └── NotificationService: send_confirmation
```

## Google Drive (облачное хранилище)

### Требования

**Функциональные:**
- Загрузка и скачивание файлов любого размера
- Синхронизация между устройствами
- Версионирование файлов
- Шаринг с другими пользователями

**Нефункциональные:**
- 1B пользователей, средний объём 15 GB на пользователя
- Большие файлы (видео до 10 GB)
- Синхронизация при конкурентном редактировании

### Разделение: метаданные vs содержимое

Ключевое архитектурное решение — файлы и их метаданные хранятся раздельно:

```
Metadata DB (PostgreSQL):
  files: { file_id, owner_id, name, size, created_at, updated_at, parent_folder_id }
  chunks: { chunk_id, file_id, chunk_index, checksum, storage_path }
  versions: { version_id, file_id, created_at, size }

File Storage (S3 / object storage):
  Содержимое чанков: s3://drive-bucket/{chunk_id}
```

### Chunked Upload (чанковая загрузка)

Загружать файл целиком — плохо: прервётся на 9 GB из 10 → начинай сначала. Решение — разбить файл на чанки (например, 4 MB каждый):

```
Клиент:
  1. Разбить файл на chunks по 4 MB
  2. Вычислить SHA-256 каждого чанка
  3. POST /upload/init → получить upload_id
  4. Для каждого чанка: PUT /upload/{upload_id}/chunk/{index}
  5. POST /upload/{upload_id}/complete

Сервер при получении чанка:
  - Проверить checksum
  - Сохранить в S3
  - Записать в chunks таблицу
  - Вернуть ACK

Прерывание → resume: клиент спрашивает "какие чанки уже есть?" → досылает только недостающие
```

**Дедупликация:** если чанк с таким же SHA-256 уже в хранилище — не загружаем повторно, просто добавляем ссылку. Экономия места при одинаковых файлах у разных пользователей.

### Delta Sync (синхронизация изменений)

При изменении файла не перезагружаем весь файл, только изменённые чанки:

```
Файл изменён:
  Старый: [chunk1][chunk2][chunk3][chunk4]
  Новый:  [chunk1][chunk2'][chunk3][chunk4]  ← изменён только chunk2

Delta sync: загружаем только chunk2' → экономия трафика
```

### Разрешение конфликтов

Два пользователя редактируют один файл офлайн → оба синхронизируются. Подходы:

- **Last-Write-Wins** — побеждает последняя по времени версия. Просто, но одно изменение теряется.
- **Conflict copy** — создаём обе версии: `document.docx` и `document (John's conflicted copy 2024-01-15).docx`. Пользователь сам разбирается. Именно так делает Dropbox.
- **OT/CRDT** — операционные преобразования для текстовых файлов (Google Docs). Сложно, но позволяет мерджить изменения автоматически.

### Синхронизация между устройствами

```
Устройство A изменило файл:
  1. Upload delta → Metadata DB обновлена
  2. File Change Event → Kafka → Sync Service
  3. Sync Service смотрит: у каких устройств этот файл подписан?
  4. Push notification (WebSocket / long polling) → устройству B
  5. Устройство B скачивает дельту

Long-polling для синхронизации:
  Клиент: GET /changes?since=last_sync_token → ждёт до 30 сек
  Сервер: при изменении отвечает списком изменений с новым sync_token
```

### Архитектура

```
Client → API Gateway
              ├── Metadata Service → PostgreSQL (файлы, папки, шаринг)
              ├── Upload Service   → S3 (содержимое)
              ├── Sync Service     → Kafka (события изменений)
              └── CDN              ← раздача популярных файлов
```

## Вопросы с собеса

**Как предотвратить двойное бронирование при высокой конкурентности?**

<details>
<summary>Ответ</summary>

Три уровня защиты:

1. **БД-уровень:** `UNIQUE constraint` на `(room_id, check_in, check_out)` — последняя линия обороны, гарантирует что дубль физически невозможен
2. **Приложение:** оптимистичная блокировка через `version` поле — `UPDATE ... WHERE version=X`, если `affected_rows=0` — конфликт, отвечаем ошибкой
3. **Кэш:** Redis `SETNX lock:room:{id}:{date}` — быстрая проверка до обращения в БД, снижает нагрузку

Для очень горячих объектов (последний номер в новогоднюю ночь) — пессимистичная блокировка `SELECT FOR UPDATE` надёжнее, хотя и снижает параллелизм.

</details>

**Зачем делать Chunked Upload и как реализовать resume?**

<details>
<summary>Ответ</summary>

**Зачем:** большой файл (10 GB) загружается часами. Любой обрыв сети требует начать заново. Чанки решают: при обрыве теряем максимум последний чанк (4 MB).

**Resume:**
```
Клиент при возобновлении:
  GET /upload/{upload_id}/status
  → сервер возвращает: { completed_chunks: [0,1,3], missing: [2,4,5,...] }
  → клиент досылает только missing chunks
```

Идентификация чанка по SHA-256 checksum: если клиент пересчитал чанк и хеш совпадает с уже загруженным — пропускаем. Это же даёт дедупликацию бесплатно.

</details>

**Как синхронизировать файл между 10 устройствами пользователя?**

<details>
<summary>Ответ</summary>

**Push-модель через WebSocket/SSE:**
1. Каждое устройство держит постоянное соединение с Sync Service
2. При изменении файла → событие в Kafka
3. Sync Service рассылает уведомление на все онлайн-устройства пользователя
4. Устройство запрашивает дельту и скачивает изменённые чанки

**Для офлайн-устройств:**
- При подключении устройство отправляет свой `last_sync_token`
- Сервер возвращает все изменения с того момента
- Изменения в Kafka хранятся с retention (например, 7 дней)

Синхронизация атомарна на уровне чанков, не файлов — частично синхронизированный файл не открывается до завершения.

</details>

**Почему поиск отелей через Elasticsearch, а не PostgreSQL?**

<details>
<summary>Ответ</summary>

Поиск отелей — это одновременно: full-text поиск по названию/описанию + геопоиск (отели в радиусе 5 км) + фильтры (рейтинг, удобства, цена) + сортировка по релевантности.

PostgreSQL может всё это, но медленно при 500k+ отелей:
- Full-text: `GIN`-индекс, но не такой гибкий как ES
- Геопоиск: `PostGIS`, работает, но медленнее ES geo_distance
- Фасетные фильтры (facets): ES нативно считает агрегации

ES оптимизирован под именно такие сценарии: инвертированные индексы для текста, BKD-дерево для геопространственных запросов, встроенные агрегации для фасетов.

PostgreSQL остаётся главным хранилищем (source of truth), ES — проекция для поиска, синхронизируется через CDC.

</details>
