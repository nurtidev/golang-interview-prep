# Урок 8: Проектирование Яндекс.Такси и Leetcode

## Яндекс.Такси (ride-hailing)

### Требования

**Функциональные:**
- Пассажир запрашивает поездку с точкой A → B
- Система находит ближайшего свободного водителя и матчит их
- Real-time отслеживание местоположения водителя
- Расчёт стоимости, оплата

**Нефункциональные:**
- 10M поездок в день, пики — утро/вечер рабочих дней
- Матчинг < 5 секунд
- Обновление геопозиции водителя каждые 3–5 секунд
- Точность геолокации: сотни метров достаточно

### Обновление геопозиций: самая нагруженная часть

1M активных водителей × обновление каждые 4 сек = **250K write/сек**. Это чистая write-heavy нагрузка.

Для хранения текущего местоположения нужно быстрое хранилище с геоиндексом — **Redis с GEO-командами**:

```
Водитель обновляет позицию:
  GEOADD drivers:active 37.617 55.755 "driver:42"  ← longitude, latitude, member

Найти водителей рядом с пассажиром:
  GEORADIUS drivers:active 37.620 55.750 3 km ASC COUNT 10
  → [driver:42, driver:17, driver:99, ...]   ← ближайшие, отсортированные по расстоянию
```

Redis GEO под капотом — это Sorted Set с геохешем в качестве score. GEORADIUS работает за O(N+log M), где N — результаты в радиусе.

### Геохеширование (Geohash)

Карта делится на прямоугольные ячейки разного размера. Каждая ячейка имеет строковый код. Соседние ячейки имеют общий префикс:

```
Точность:
  Geohash 4 символа: ~40x20 км
  Geohash 6 символов: ~1.2x0.6 км  ← достаточно для такси
  Geohash 8 символов: ~38x19 м

Поиск водителей рядом:
  Пассажир в ячейке "ucfv0e"
  → ищем водителей в "ucfv0e" и 8 соседних ячейках
  → нет водителей → расширяем до родительской ячейки "ucfv0" (6 соседей)
```

### Алгоритм матчинга

Простая версия: берём N ближайших водителей, ранжируем (расстояние, рейтинг, время ожидания), предлагаем первому. Не взял (таймаут 5 сек) → следующему.

```
Passenger requests ride:
  1. Location Service: GEORADIUS → [driver1, driver2, driver3]
  2. Filter: только свободные (статус = AVAILABLE)
  3. Sort: по расстоянию + rating
  4. Offer to driver1 → wait 5 sec
     → accept: создать поездку, обновить статус обоих
     → timeout/decline: offer to driver2 → ...
```

### Real-time отслеживание

Пассажир видит водителя на карте в реальном времени. Нужен push каждые 3–5 секунд:

```
Водитель → Location Update → Location Service → Redis (текущая позиция)
                                              → Kafka → Tracking Service

Пассажир: WebSocket соединение с Tracking Service
  Tracking Service подписан на Kafka topic driver:{driver_id}
  При новом событии → push через WebSocket → обновление маркера на карте
```

### Расчёт цены (surge pricing)

Цена динамически меняется в зависимости от спроса и предложения в конкретном районе:

```
Demand: количество активных заказов в geohash-ячейке за последние 5 минут
Supply: количество свободных водителей в ячейке
Surge multiplier = f(demand / supply)

Пример: demand=100, supply=10 → multiplier=2.5x
```

### Архитектура

```
Driver App     → Location Service  → Redis GEO + Kafka
Passenger App  → Matching Service  ← Redis GEO (поиск водителей)
                     ↓
               Trip Service → PostgreSQL (поездки, история)
                     ↓
               Payment Service → внешний платёжный шлюз
                     ↓
               Notification Service → Push (APNs/FCM)
```

## Leetcode (онлайн-судья)

### Требования

**Функциональные:**
- Пользователь пишет решение задачи на одном из языков (Go, Python, Java, ...)
- Система запускает код против набора тест-кейсов
- Возвращает результат: Accepted / Wrong Answer / TLE / Runtime Error

**Нефункциональные:**
- Безопасное выполнение кода (sandboxing — пользователь не может навредить системе)
- Таймаут выполнения (обычно 1–2 секунды)
- Честные лимиты памяти
- 100K submissions/день, пики — во время контестов

### Главная проблема: безопасное выполнение чужого кода

Пользовательский код может делать что угодно: fork-bomb, чтение /etc/passwd, выход в сеть, бесконечный цикл. Нужна изоляция.

**Уровни изоляции:**

| Метод | Изоляция | Overhead | Сложность |
|-------|----------|----------|-----------|
| Process + seccomp | средняя | низкий | средняя |
| Namespace (cgroups) | хорошая | низкий | высокая |
| Docker container | хорошая | средний | низкая |
| gVisor / Firecracker | отличная | высокий | высокая |

**На практике используют комбинацию:**
- Docker-контейнер с ограниченными ресурсами (CPU, память)
- seccomp-фильтр: блокируем системные вызовы (fork, exec, network)
- `cgroups`: жёсткий лимит CPU-времени и RAM
- Отдельная сеть без доступа в интернет
- Read-only файловая система

```
Submission → Judge Worker
                 ↓
         Docker container (per submission):
           - time limit: 1000ms wall clock
           - memory: 256 MB
           - no network
           - seccomp: block fork/exec/socket
           ↓
         Run code against each test case
         Capture stdout/stderr, exit code, time, memory
```

### Асинхронная очередь для сабмитов

Компиляция + выполнение кода занимает 0.1–2 секунды. При 100K/день пиковая нагрузка может быть ~100 submission/сек. Нельзя выполнять синхронно в HTTP-запросе.

```
Клиент → POST /submit → {submission_id: "abc123", status: "PENDING"}
                ↓
           Kafka queue (submissions)
                ↓
         Judge Worker Pool (N воркеров, каждый = Docker контейнер)
                ↓
         Результат → DB + WebSocket push → Клиент видит результат
```

Клиент сразу получает `submission_id`, затем подключается через WebSocket или polling:
```
GET /submission/abc123/status → { status: "RUNNING", progress: "2/10 tests" }
                               → { status: "ACCEPTED", time: 42ms, memory: 18MB }
```

### Хранение тест-кейсов

Тест-кейсы не хранятся в БД — это большие файлы (для некоторых задач input/output по нескольку GB). Хранятся в object storage (S3), воркер скачивает при старте или кэширует локально.

```
S3: /problems/{problem_id}/
      input_1.txt, output_1.txt
      input_2.txt, output_2.txt
      ...
      input_100.txt, output_100.txt

Воркер: предзагружает тест-кейсы популярных задач в локальный кэш.
```

### Результаты и статистика

```
submissions:
  { id, user_id, problem_id, language, code, status, time_ms, memory_mb, created_at }

Индексы:
  (user_id, problem_id) — "решил ли этот пользователь эту задачу?"
  (problem_id, status, created_at) — статистика Acceptance Rate
```

### Архитектура

```
Browser → API Service → PostgreSQL (задачи, результаты)
              ↓
           Kafka (submission queue)
              ↓
         Judge Workers (autoscaling group)
              ├── Docker sandbox (выполнение)
              ├── S3 (тест-кейсы)
              └── результат → Kafka → WebSocket Service → Browser
```

**Autoscaling:** во время контеста количество сабмитов вырастает в 10-20 раз. Judge Workers масштабируются горизонтально — каждый независимый, stateless, берёт задания из Kafka.

## Вопросы с собеса

**Как обработать 250K геолокационных обновлений/сек от водителей?**

<details>
<summary>Ответ</summary>

Это write-heavy нагрузка с требованием свежести данных (< 5 сек). PostgreSQL не справится без серьёзного шардирования.

**Решение — Redis GEO:**
- В памяти: write ~100K ops/сек без проблем
- Геопространственные операции встроены: `GEOADD`, `GEORADIUS`
- TTL: водители автоматически "пропадают" из активных если перестали отправлять обновления

**Шардирование Redis по городам:** ключ `drivers:active:moscow`, `drivers:active:spb` — каждый город на своём Redis-инстансе. Матчинг происходит только в одном городе.

Для персистентности (история поездок, аналитика) — параллельно пишем в Kafka, оттуда в ClickHouse для аналитики.

</details>

**Что такое geohash и зачем он нужен в такси?**

<details>
<summary>Ответ</summary>

Geohash — кодирование географических координат в строку. Карта рекурсивно делится на прямоугольники, каждый получает буквенно-цифровой код.

Главное свойство: **пространственная близость = общий префикс**. `ucfv0e` и `ucfv0d` — соседние ячейки.

В такси используется для:
1. **Эффективный поиск соседей:** ищем в ячейке + 8 соседних ячейках по префиксу
2. **Шардирование:** каждый сервер отвечает за свои ячейки
3. **Surge pricing:** считаем спрос/предложение в ячейке, а не по всему городу
4. **Кэширование:** результаты поиска для популярных ячеек кэшируем

Альтернатива — S2 Cells (Google), более равномерное деление сферы, используется в Uber H3.

</details>

**Как изолировать выполнение пользовательского кода?**

<details>
<summary>Ответ</summary>

Несколько уровней защиты:

**Linux namespaces + cgroups:**
- `pid namespace`: процесс не видит другие процессы системы
- `net namespace`: нет доступа к сети
- `mount namespace`: read-only файловая система
- `cgroup CPU`: жёсткий лимит времени (SIGKILL при превышении)
- `cgroup memory`: лимит RAM + swap

**seccomp (syscall filter):** whitelist разрешённых системных вызовов. Блокируем: `fork`, `exec`, `socket`, `open` (опционально). Пользовательский код не может создавать процессы или открывать сокеты.

**Docker** предоставляет всё это через удобный API: `--cpus`, `--memory`, `--network none`, `--read-only`, `--security-opt seccomp=profile.json`.

**Дополнительно:** запуск от unprivileged пользователя (не root), tmpfs для временных файлов, таймаут на уровне оркестратора (not just CPU time, but wall clock time).

</details>

**Как масштабировать Judge Workers во время контеста (пик 10x)?**

<details>
<summary>Ответ</summary>

Judge Workers stateless — каждый независимо берёт задание из Kafka, выполняет, пишет результат. Это позволяет горизонтально масштабировать без координации.

**Автоскейлинг:**
1. Метрика: глубина очереди Kafka (consumer lag)
2. При lag > N — запустить новые воркеры (Kubernetes HPA по custom metrics)
3. Новый воркер: скачать образ → старт контейнера → начать потреблять из Kafka
4. После контеста lag снижается → воркеры масштабируются вниз

**Предзапуск воркеров:** за 10 минут до контеста — prewarming: запустить N готовых воркеров. Холодный старт Docker-контейнера занимает секунды, что критично под нагрузкой.

**Ограничение:** воркеры изолированы друг от друга, каждый — отдельный Docker контейнер. Нельзя выполнять два сабмита в одном контейнере параллельно.

</details>
