# Слайсы (Slices)

---

## Лекция

### Что такое слайс и зачем он нужен

В большинстве языков есть массивы — структуры фиксированного размера. Проблема: при создании массива нужно знать размер заранее. В реальных задачах это часто невозможно: мы не знаем, сколько строк в файле, сколько пользователей вернёт запрос.

Go решает это через **слайсы** — динамические представления над массивами. Слайс не хранит данные сам — он ссылается на **underlying array** (базовый массив) и знает свою длину и ёмкость.

---

### Внутреннее устройство

Слайс — это структура из трёх полей (`runtime/slice.go`):

```go
type slice struct {
    array unsafe.Pointer // указатель на первый элемент базового массива
    len   int            // количество доступных элементов
    cap   int            // ёмкость: сколько элементов помещается до конца массива
}
```

Визуально:

```
s := make([]int, 3, 5)

  slice header       underlying array
┌─────────────┐     ┌────┬────┬────┬────┬────┐
│ ptr ─────────────→│  0 │  0 │  0 │  _ │  _ │
│ len = 3     │     └────┴────┴────┴────┴────┘
│ cap = 5     │      [0]  [1]  [2]  [3]  [4]
└─────────────┘      ↑              ↑
                   len=3          cap=5
```

Когда вы передаёте слайс в функцию — передаётся **копия заголовка** (ptr + len + cap), но не копия данных. Изменения элементов через один заголовок видны через другой.

---

### Отличие слайса от массива

```go
a := [3]int{1, 2, 3}  // массив: тип [3]int, размер — часть типа
s := []int{1, 2, 3}   // слайс: тип []int, размер не фиксирован
```

| | Массив | Слайс |
|---|---|---|
| Размер | фиксирован в типе | динамический |
| Передача в функцию | копируется **целиком** | копируется только заголовок (8+8+8 байт) |
| Тип | `[3]int` ≠ `[4]int` | `[]int` — всегда |
| Сравнение `==` | ✅ (если одинаковый тип) | ❌ compile error |
| `nil` | не бывает nil | может быть nil |

```go
b := a  // полная копия данных — изменение b не затронет a
t := s  // копия заголовка — данные общие!

t[0] = 99
fmt.Println(s[0]) // 99 — s тоже "увидел" изменение
```

### append и рост capacity

`append` добавляет элементы в слайс. Если `len < cap` — просто записывает в следующую ячейку базового массива и возвращает слайс с увеличенным `len`. Если `len == cap` — нужно выделить новый, больший массив.

```
append когда cap хватает:

ПЕРЕД:  ptr → [1, 2, 3, _, _]   len=3, cap=5
ПОСЛЕ:  ptr → [1, 2, 3, 4, _]   len=4, cap=5
(тот же массив, просто len++)
```

```
append когда cap не хватает:

ПЕРЕД:  ptr → [1, 2, 3]          len=3, cap=3
ПОСЛЕ:  ptr → [1, 2, 3, 4, _, _, ...]  len=4, cap≈6
(новый массив, данные скопированы)
```

**Стратегия роста capacity (приблизительно):**
- Если `cap < 256` — удваивается
- Если `cap >= 256` — растёт на ~25% + поправка
- Точная формула менялась в версиях Go (1.18 изменила подход)

```go
s := []int{1, 2, 3}           // len=3, cap=3
s = append(s, 4)               // cap недостаточно → новый массив
                               // len=4, cap≈6
```

**Критически важно:** после `append` с ростом старый и новый слайс указывают на **разные** массивы. Изменения одного не видны в другом.

### Опасная ловушка: общий базовый массив

Когда capacity хватает — новый массив не создаётся, и два слайса делят один:

```go
a := make([]int, 3, 6) // len=3, cap=6
b := a                  // b и a — разные заголовки, но один массив

b = append(b, 99)       // cap хватает → пишем в позицию [3] общего массива
                        // b.len=4, a.len=3

// Через заголовок a позиция [3] недостижима (len=3)
// Но если сделать a = a[:4]:
a2 := a[:4]
fmt.Println(a2[3]) // 99 — данные там!
```

**Классическая ловушка на собеседовании:**
```go
s := make([]int, 0, 5)
a := append(s, 1, 2, 3) // a → [1, 2, 3], s.len=0 (s "не знает" об a)
b := append(s, 4, 5, 6) // b пишет с позиции 0 того же массива!

fmt.Println(a) // [4 5 6] — НЕ [1 2 3]! b перезаписал данные
fmt.Println(b) // [4 5 6]
```

Почему? `s` имеет `len=0, cap=5`. Оба `append(s, ...)` видят одну и ту же "пустую" часть capacity и пишут с начала.

### Нарезка (slicing) и трёх-индексная форма

```go
arr := [5]int{1, 2, 3, 4, 5}

s1 := arr[1:3]   // len=2, cap=4, данные: [2, 3]
                  // cap=4 потому что до конца arr: [2,3,4,5]

s2 := arr[1:3:4] // len=2, cap=3 (три-индексная форма: arr[low:high:max])
                  // cap ограничен: [2,3,4] — не дотягивается до [5]
```

Трёх-индексная форма `[low:high:max]` ограничивает capacity. Это важно чтобы предотвратить запись за пределы "логически выделенного" диапазона.

### copy — независимая копия

`copy(dst, src)` копирует `min(len(dst), len(src))` элементов и возвращает количество скопированных. Dst и src не делят память.

```go
src := []int{1, 2, 3}
dst := make([]int, len(src))
n := copy(dst, src)
dst[0] = 99
fmt.Println(src[0]) // 1 — не изменился
```

### nil слайс vs пустой слайс

```go
var s1 []int          // nil слайс:   ptr=nil, len=0, cap=0, s1 == nil → true
s2 := []int{}         // пустой слайс: ptr≠nil, len=0, cap=0, s2 == nil → false
s3 := make([]int, 0)  // пустой слайс: s3 == nil → false
```

Практически — оба ведут себя одинаково: `append`, `range`, `len`, `cap` работают корректно.

**НО — важное отличие в JSON:**
```go
json.Marshal(s1) // → null
json.Marshal(s2) // → []
```

Правило: возвращай `nil` слайс когда "нет данных", и пустой `[]T{}` когда "данные есть, но их 0" (например, успешный запрос, вернувший 0 результатов).

### Утечки памяти через слайсы

Слайс держит ссылку на весь базовый массив — GC не освободит его пока живёт хотя бы один слайс, ссылающийся на этот массив.

```go
// ❌ Утечка: возвращаем маленький слайс, но держим огромный массив в памяти
func firstThreeBytes(data []byte) []byte {
    return data[:3] // underlying array — весь data — не освобождается!
}

// ✅ Правильно: копируем только нужное
func firstThreeBytes(data []byte) []byte {
    result := make([]byte, 3)
    copy(result, data[:3])
    return result
    // data теперь может быть собран GC
}
```

Типичный сценарий: читаем большой файл в `[]byte`, берём маленький кусок — держим всё в памяти.

### range по слайсу

```go
s := []int{10, 20, 30}

for i, v := range s {
    fmt.Println(i, v) // v — копия элемента, не ссылка
}

// Длина вычисляется ОДИН РАЗ в начале range
for i := range s {
    s = append(s, 0) // добавляем, но итераций всё равно 3 (изначальный len=3)
}
```

Важно: `v` в `for i, v := range` — это **копия** значения. Изменение `v` не меняет слайс. Чтобы изменить — используй `s[i]`.

## Примеры

### Пример 1: Общий базовый массив — наблюдаем побочный эффект

```go
package main

import "fmt"

func main() {
    original := []int{1, 2, 3, 4, 5}
    sub := original[1:4] // [2, 3, 4], len=3, cap=4

    fmt.Println("до изменения:", original) // [1 2 3 4 5]

    sub[0] = 99 // изменяем через sub

    fmt.Println("после изменения sub[0]=99:")
    fmt.Println("original:", original) // [1 99 3 4 5] — изменился!
    fmt.Println("sub:     ", sub)      // [99 3 4]

    // append в пределах capacity: перезапишет original[4]
    sub = append(sub, 777)
    fmt.Println("после append в sub:")
    fmt.Println("original:", original) // [1 99 3 4 777] — снова изменился!
}
```

### Пример 2: copy для защиты от побочных эффектов

```go
package main

import "fmt"

func modifyFirst(s []int) []int {
    // Работаем с копией — original не затронут
    result := make([]int, len(s))
    copy(result, s)
    result[0] = 999
    return result
}

func main() {
    original := []int{1, 2, 3}
    modified := modifyFirst(original)

    fmt.Println("original:", original) // [1 2 3] — не изменился
    fmt.Println("modified:", modified) // [999 2 3]
}
```

### Пример 3: Ловушка с append и общим capacity

```go
package main

import "fmt"

func main() {
    base := make([]int, 0, 5)

    // Оба append видят одну и ту же "пустую" часть
    a := append(base, 1, 2, 3)
    b := append(base, 7, 8, 9) // перезаписывает те же позиции!

    fmt.Println("a:", a) // [7 8 9] — а не [1 2 3]!
    fmt.Println("b:", b) // [7 8 9]

    // Как избежать: использовать трёх-индексную форму для ограничения cap
    base2 := make([]int, 0, 5)[:0:0] // cap=0 → append создаст новый массив
    c := append(base2, 1, 2, 3)
    d := append(base2, 7, 8, 9)

    fmt.Println("c:", c) // [1 2 3]
    fmt.Println("d:", d) // [7 8 9] — теперь независимы
}
```

### Пример 4: Трёх-индексная нарезка как защита

```go
package main

import "fmt"

func process(data []int) []int {
    // Возвращаем срез с ограниченным cap
    // Вызывающий код не сможет случайно залезть за границы нашего среза
    return data[1:3:3] // len=2, cap=2 (не до конца data!)
}

func main() {
    data := []int{10, 20, 30, 40, 50}
    result := process(data)

    fmt.Println("result:", result) // [20 30]
    fmt.Println("len:", len(result), "cap:", cap(result)) // 2, 2

    // append создаст новый массив (cap исчерпан)
    result = append(result, 99)
    fmt.Println("data после append:", data) // [10 20 30 40 50] — не затронут
}
```

### Пример 5: Утечка памяти и исправление

```go
package main

import (
    "fmt"
    "runtime"
)

// ❌ Утечка: держим гигантский буфер ради 3 байт
func leakyExtract(bigData []byte) []byte {
    return bigData[:3] // GC не освободит bigData
}

// ✅ Правильно: делаем независимую копию
func safeExtract(bigData []byte) []byte {
    result := make([]byte, 3)
    copy(result, bigData[:3])
    return result
    // bigData теперь может быть собран GC если нет других ссылок
}

func main() {
    bigData := make([]byte, 10*1024*1024) // 10 МБ
    bigData[0], bigData[1], bigData[2] = 'G', 'o', '!'

    extracted := safeExtract(bigData)
    bigData = nil // явно убираем ссылку
    runtime.GC()

    fmt.Println(string(extracted)) // Go!
}
```

## Вопросы с собеседования

**Q: Что выведет этот код?**
```go
s := []int{1, 2, 3}
s2 := s[1:]
s2[0] = 99
fmt.Println(s)
```

`[1 99 3]`. `s2` и `s` делят один базовый массив. `s2[0]` — это `s[1]`. Изменение через `s2` видно через `s`.

---

**Q: Что выведет?**
```go
s := make([]int, 3)
for i, v := range s {
    s = append(s, i+v)
}
fmt.Println(len(s))
```

`6`. `range` вычисляет длину один раз при старте — `len=3`. Цикл делает 3 итерации: добавляем 3 элемента. Итого `len=6`. То, что `s` растёт внутри цикла, на количество итераций не влияет.

---

**Q: Что такое cap и зачем он нужен?**

`cap` — ёмкость: сколько элементов поместится до конца базового массива начиная с позиции `ptr`. Нужен для оптимизации `append`: если `len < cap`, не нужно выделять новый массив. Правило хорошего тона: если знаешь примерный размер заранее — передавай hint в `make([]T, 0, n)` чтобы избежать лишних аллокаций.

---

**Q: В чём разница между nil слайсом и пустым слайсом?**

`nil` слайс: `var s []int` — ptr=nil, len=0, cap=0, `s == nil` → true.
Пустой слайс: `s := []int{}` — ptr≠nil, len=0, cap=0, `s == nil` → false.

Практически они одинаковы: `len`, `cap`, `append`, `range` работают корректно для обоих. Разница: `json.Marshal(nil)` → `null`, `json.Marshal([]int{})` → `[]`. Это важно в API-ответах.

---

**Q: Когда append создаёт новый массив, а когда нет?**

Если `len < cap` — новый массив не создаётся, данные пишутся в существующий. Если `len == cap` — выделяется новый массив (примерно вдвое больше для малых слайсов), данные копируются, старый массив GC освободит когда не будет ссылок. Поэтому после `append` всегда нужно переприсваивать: `s = append(s, ...)`.

---

**Q: Как скопировать слайс так, чтобы изменения в копии не затрагивали оригинал?**

```go
dst := make([]int, len(src))
copy(dst, src)
```

Или короче: `dst := append([]int{}, src...)`. Оба способа создают независимую копию данных.

---

**Q: Что такое утечка памяти через слайс?**

Если маленький слайс является подсрезом большого массива — GC не освободит этот массив, пока живёт маленький слайс. Решение: `copy` — создать независимый слайс нужного размера, скопировать данные. Тогда большой массив станет недостижимым и GC его освободит.

---

**Q: Что выведет?**
```go
s := make([]int, 0, 5)
a := append(s, 1, 2, 3)
b := append(s, 4, 5, 6)
fmt.Println(a)
fmt.Println(b)
```

Оба выведут `[4 5 6]`. Потому что `s` имеет `len=0, cap=5`. Оба `append(s, ...)` начинают запись с позиции 0 одного и того же базового массива. `b` перезаписывает данные, записанные `a`. Чтобы избежать — использовать `a = append(s[:len(s):len(s)], 1, 2, 3)` (ограничиваем cap).
