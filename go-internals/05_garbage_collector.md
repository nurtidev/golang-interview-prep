# Garbage Collector в Go

---

## Лекция

### 1. Зачем нужен GC и что он делает

В языках с ручным управлением памятью (C, C++) программист сам выделяет (`malloc`) и освобождает (`free`) память. Забыл освободить — утечка памяти. Освободил дважды — краш. Освободил и продолжил использовать — undefined behavior.

Go использует **автоматический сборщик мусора**: программист выделяет память, GC сам определяет когда она больше не нужна и освобождает.

**Задача GC:** найти объекты, на которые нет ни одной живой ссылки, и вернуть их память операционной системе (или пулу).

---

### 2. Стек vs Куча — где живут объекты

Не все объекты попадают под GC. Go использует **escape analysis** (анализ выхода) на этапе компиляции:

```
Объект остаётся на стеке:
- локальная переменная, которая не уходит за пределы функции
- стек быстрее (нет GC оверхеда), освобождается автоматически при возврате из функции

Объект уходит на кучу (heap):
- возвращаем указатель на локальную переменную
- переменная захвачена замыканием
- размер неизвестен на этапе компиляции
- объект слишком большой
```

```go
// Остаётся на стеке — не нагружает GC
func sum(a, b int) int {
    result := a + b  // result на стеке
    return result
}

// Уходит на кучу — GC будет следить
func newUser(name string) *User {
    u := User{Name: name}  // u уходит на heap — возвращаем указатель
    return &u
}
```

Проверить escape analysis:
```bash
go build -gcflags="-m" ./...
# выведет: "moved to heap: u"
```

### 3. Алгоритм: Tri-color Mark and Sweep

Go использует **трёхцветный алгоритм пометки и очистки** (tri-color mark-and-sweep) с параллельным выполнением.

**Три множества объектов:**
```
Белые (White) — кандидаты на удаление (не посещены)
Серые  (Grey)  — посещены, но их ссылки ещё не проверены
Чёрные (Black) — посещены и все их ссылки проверены
```

**Фазы работы GC:**

```
1. STW (Stop The World) — короткая пауза ~0.1ms
   └── Scan roots: стеки горутин, глобальные переменные → помечаем серыми

2. Mark (параллельно с программой) — основная фаза
   └── Берём серый объект
       └── Помечаем его чёрным
           └── Его ссылки помечаем серыми
   Повторяем пока серых не останется

3. STW — ещё одна короткая пауза
   └── Финализация: проверяем что ничего не пропустили

4. Sweep (параллельно с программой)
   └── Все белые объекты → недостижимы → освобождаем память
```

**Инвариант:** чёрный объект никогда не ссылается напрямую на белый.

---

### 4. Write Barrier (барьер записи)

**Проблема параллельной маркировки:**

```
GC пометил объект A чёрным (все ссылки проверены).
Программа выполняется параллельно и делает:
    A.ref = C   // C был белым (не посещён)
GC не знает о новой ссылке → C удалится → краш!
```

**Решение — Write Barrier:** при каждом изменении указателя (пока идёт фаза Mark) — автоматически добавляем объект в серое множество.

```go
// Компилятор вставляет barrier автоматически:
A.ref = C
// превращается примерно в:
writeBarrier(A, C)  // пометить C серым если GC активен
A.ref = C
```

Go использует **Hybrid Write Barrier** (Go 1.14+) — комбинация Dijkstra + Yuasa барьеров. Позволяет не останавливать горутины для re-scan стеков.

### 5. GOGC и триггер запуска GC

GC не запускается постоянно. Триггер — `GOGC` (по умолчанию **100**):

```
GOGC=100 означает: запустить GC когда heap вырос на 100% от размера после прошлого GC

После GC heap = 10 MB
→ следующий GC запустится при heap = 20 MB

GOGC=50  → агрессивнее, чаще GC, меньше памяти, больше CPU
GOGC=200 → реже GC, больше памяти, меньше CPU
GOGC=off → отключить GC (опасно!)
```

```go
import "runtime/debug"

debug.SetGCPercent(50)  // изменить GOGC в рантайме
```

**Go 1.19+: GOMEMLIMIT** — лимит памяти для GC:
```go
debug.SetMemoryLimit(512 * 1024 * 1024) // 512 MB
```
GC начнёт агрессивнее работать когда приближается к лимиту. Помогает избежать OOM killer.

### 6. Паузы GC (STW)

Исторически GC паузы были проблемой. Эволюция:

```
Go 1.4:  STW паузы ~ 300ms (для больших heap)
Go 1.5:  concurrent GC — параллельная маркировка → паузы < 10ms
Go 1.8:  убран re-scan горутин-стеков → паузы < 1ms
Go 1.14: Hybrid Write Barrier (Dijkstra + Yuasa) → паузы ~ 0.1ms (100 микросекунд)
Go 1.26:  Green Tea GC — 10–40% снижение GC-оверхеда (по умолчанию)
```

Сейчас паузы настолько малы, что для большинства задач незаметны. Проблема актуальна только для latency-критичных систем (HFT, real-time).

**Green Tea GC (Go 1.26, по умолчанию):** новая реализация сборщика мусора, улучшающая маркировку и сканирование мелких объектов через лучшую локальность и CPU-масштабируемость. Не меняет алгоритм (tri-color mark-and-sweep), но существенно снижает overhead. Отключить: `GOEXPERIMENT=nogreenteagc`.

### 7. Finalizers и runtime.GC()

```go
// Finalizer — функция вызывается перед удалением объекта GC
obj := &Resource{}
runtime.SetFinalizer(obj, func(r *Resource) {
    r.Close() // освободить внешний ресурс
})

// Go 1.24+: AddCleanup — более гибкая и безопасная альтернатива SetFinalizer
// ВАЖНО: arg не должен быть самим obj — иначе GC никогда не соберёт объект
handle := obj.handle // извлекаем ресурс отдельно
runtime.AddCleanup(obj, func(h *Handle) {
    h.Close()
}, handle)
// Преимущества перед SetFinalizer:
// - можно добавить несколько cleanup-функций на один объект
// - cleanup не сбрасывается при SetFinalizer на том же объекте
// - менее подвержен ошибкам использования (не возрождает объект)

// Принудительный запуск GC (для тестов, не для прода)
runtime.GC()
```

**Не злоупотребляй finalizers/cleanup:**
- Нет гарантии когда вызовется
- Нет гарантии что вызовется вообще (при завершении программы)
- Используй `defer r.Close()` вместо finalizer
- Предпочитай `runtime.AddCleanup` над `runtime.SetFinalizer` в Go 1.24+

### 8. Практика: как снизить нагрузку на GC

**Переиспользуй объекты через sync.Pool:**
```go
var bufPool = sync.Pool{
    New: func() any {
        return make([]byte, 0, 4096)
    },
}

func handler() {
    buf := bufPool.Get().([]byte)
    defer bufPool.Put(buf[:0]) // вернуть в пул очищенным

    // используем buf...
}
```
`sync.Pool` — кэш объектов между GC циклами. GC может очистить пул, но это нормально.

**Избегай аллокаций в hot path:**
```go
// Плохо — каждый вызов аллоцирует строку на heap
func key(userID int) string {
    return fmt.Sprintf("user:%d", userID)
}

// Лучше — переиспользуем буфер
func key(buf []byte, userID int) []byte {
    return strconv.AppendInt(buf[:0], int64(userID), 10)
}
```

**Используй value types вместо pointer types когда возможно:**
```go
// Pointer — уходит на heap, нагружает GC
users := []*User{...}

// Value — может остаться на стеке или в contiguous массиве
users := []User{...}
```

**Профилирование аллокаций:**
```bash
go test -bench=. -memprofile=mem.out
go tool pprof mem.out
# top10 — топ аллокаций
# list FuncName — построчно
```

## Вопросы на собеседовании

### Q: Как работает GC в Go?

**Ответ:** Go использует трёхцветный алгоритм mark-and-sweep с параллельным выполнением. Объекты делятся на белые (не посещены), серые (посещены, ссылки не проверены), чёрные (проверены полностью). GC начинает с корней (стеки горутин, глобальные переменные), итеративно обходит граф объектов. Белые объекты в конце — мусор, освобождаются. Работает параллельно с программой, STW паузы < 1ms.

---

### Q: Что такое Write Barrier и зачем он нужен?

**Ответ:** Write Barrier — автоматически вставляемый компилятором код при изменении указателей во время фазы маркировки. Нужен потому что GC работает параллельно с программой: без барьера программа могла бы создать новую ссылку от уже проверенного (чёрного) объекта на непроверенный (белый), и GC удалил бы живой объект. Барьер перехватывает такое присваивание и помечает целевой объект серым.

---

### Q: Что такое escape analysis?

**Ответ:** Анализ выхода — оптимизация компилятора, определяющая где жить объекту: на стеке или куче. Стек быстрее (нет GC), но объект должен не "выходить" за пределы функции. Если возвращаем указатель на локальную переменную, захватываем в замыкание или размер неизвестен — объект уходит на кучу под управление GC. Проверяется через `go build -gcflags="-m"`.

---

### Q: Что такое GOGC? Как настроить GC?

**Ответ:** GOGC — процент роста heap для запуска GC (по умолчанию 100: запустить когда heap удвоился). Меньше значение — чаще GC, меньше память, больше CPU. Больше — реже GC, больше памяти. С Go 1.19 есть `GOMEMLIMIT` — абсолютный лимит памяти, GC работает агрессивнее при приближении к лимиту. Для большинства сервисов менять GOGC не нужно.

---

### Q: Как уменьшить нагрузку на GC?

**Ответ:** Три основных подхода: 1) `sync.Pool` — переиспользовать объекты вместо повторных аллокаций (буферы, временные структуры). 2) Избегать аллокаций в hot path — использовать `strconv.AppendInt` вместо `fmt.Sprintf`, value types вместо pointer types там где возможно. 3) Профилировать через `go test -memprofile` + `pprof` — найти где больше всего аллокаций и оптимизировать целенаправленно.
