# Память в Go: Стек и Куча

---

## Лекция

### 1. Два места где живут данные

В Go каждая переменная живёт либо на **стеке**, либо на **куче (heap)**.

```
Горутина 1          Горутина 2
┌──────────┐        ┌──────────┐
│  Stack   │        │  Stack   │        ← у каждой горутины свой стек
│  (2 KB)  │        │  (2 KB)  │
└──────────┘        └──────────┘

┌──────────────────────────────┐
│            Heap              │        ← одна куча на всю программу
│   (управляется GC)           │
└──────────────────────────────┘
```

---

### 2. Стек (Stack)

Стек — это область памяти для **локальных переменных функции**. Работает по принципу LIFO.

```
main() вызывает foo() вызывает bar()

Stack:
┌─────────────┐  ← top
│  bar() frame │  локальные переменные bar
├─────────────┤
│  foo() frame │  локальные переменные foo
├─────────────┤
│ main() frame │  локальные переменные main
└─────────────┘
```

Когда функция возвращается — её фрейм **мгновенно уничтожается** (просто двигаем указатель). Никакого GC не нужно.

**Стек в Go — growable (растущий):**

В большинстве языков стек фиксированного размера (обычно 1-8 MB на поток). В Go стек горутины начинается с **2-8 KB** и растёт по необходимости.

```
Горутина создаётся → стек 2 KB
Функции вызываются глубже → стек растёт → 4 KB → 8 KB → ...
Функции возвращаются → стек сжимается обратно
```

Именно поэтому можно запустить **миллион горутин** — каждая начинает с 2 KB, а не с 1 MB как поток ОС.

**Как Go увеличивает стек (copying stack):**

```
1. Горутина исчерпала текущий стек
2. Go аллоцирует новый стек в 2 раза больше
3. Копирует всё содержимое старого стека в новый
4. Обновляет все указатели
5. Освобождает старый стек
```

Это называется **copying stack** (в отличие от старого подхода — segmented stack, где стек был из отдельных сегментов).

---

### 3. Куча (Heap)

Куча — общая область памяти для объектов чей **жизненный цикл не ограничен одной функцией**.

```go
func newUser() *User {
    u := User{Name: "Алибек"}
    return &u   // u уходит на heap — переживёт эту функцию
}
```

Объекты на куче живут пока на них есть хотя бы одна ссылка. Когда ссылок нет — GC их удаляет.

**Куча медленнее стека:**
- Аллокация требует поиска свободного блока
- GC должен следить за каждым объектом
- Фрагментация памяти

---

### 4. Escape Analysis — кто решает где жить объекту

**Escape analysis** — анализ компилятора который решает: объект остаётся на стеке или "убегает" на кучу.

Объект **убегает на heap** если:

**1. Возвращаем указатель на локальную переменную:**
```go
func foo() *int {
    x := 42
    return &x   // x убегает на heap — после возврата функции стек исчезнет
}
```

**2. Переменная захвачена замыканием:**
```go
func counter() func() int {
    count := 0          // count убегает на heap
    return func() int {
        count++         // замыкание живёт дольше функции counter
        return count
    }
}
```

**3. Передаём в интерфейс:**
```go
func log(v any) { ... }

x := 42
log(x)   // x может убежать на heap — компилятор не знает что делает log()
```

**4. Размер неизвестен на этапе компиляции:**
```go
s := make([]int, n)   // n неизвестно → heap
```

**5. Объект слишком большой:**
```go
var arr [1_000_000]int   // слишком большой для стека → heap
```

**Проверить escape analysis:**
```bash
go build -gcflags="-m" ./...

# Вывод:
# ./main.go:5:6: moved to heap: x
# ./main.go:12:9: count escapes to heap
```

### 5. Аллокатор памяти Go (TCMalloc)

Go использует аллокатор на основе **TCMalloc** (Thread-Caching Malloc от Google). Три уровня:

```
┌──────────────────────────────────────────┐
│  mcache  (per-P, без блокировок)         │  ← горячий кэш, нет lock
├──────────────────────────────────────────┤
│  mcentral (per size class, с блокировкой)│  ← резерв для mcache
├──────────────────────────────────────────┤
│  mheap   (глобальный, с блокировкой)     │  ← берёт у ОС через mmap
└──────────────────────────────────────────┘
```

**mcache** — каждый логический процессор (P в GMP) имеет свой кэш. Аллокация маленьких объектов (<32 KB) происходит **без блокировок** — просто берём из своего mcache.

**Size classes** — память разбита на классы размеров (8B, 16B, 32B, ... 32KB). Объект кладётся в ближайший подходящий класс:
```
объект 20 байт → класс 24 байта (небольшой oверхед, зато нет фрагментации)
```

**Большие объекты** (>32 KB) — аллоцируются напрямую из mheap.

---

### 6. Стек vs Heap — сравнение

| | Стек | Куча |
|--|------|------|
| Скорость аллокации | Мгновенно (сдвиг указателя) | Медленнее (поиск блока) |
| Освобождение | Автоматически при возврате | GC |
| Размер | Начинает с 2 KB, растёт | Ограничен RAM |
| Потокобезопасность | Только своя горутина | Общая, нужна синхронизация |
| Жизненный цикл | Ограничен функцией | До последней ссылки |

### 7. Практика: как уменьшить аллокации на heap

**Возвращай значение вместо указателя (если объект маленький):**
```go
// Плохо — User убегает на heap
func newUser(name string) *User {
    return &User{Name: name}
}

// Лучше — User остаётся на стеке вызывающей функции
func newUser(name string) User {
    return User{Name: name}
}
```

**Передавай буфер снаружи вместо создания внутри:**
```go
// Плохо — каждый вызов аллоцирует новый слайс
func process() []byte {
    buf := make([]byte, 1024)
    // ...
    return buf
}

// Лучше — вызывающий управляет памятью
func process(buf []byte) []byte {
    // используем переданный buf
    return buf[:n]
}
```

**sync.Pool для часто создаваемых объектов:**
```go
var pool = sync.Pool{
    New: func() any { return &bytes.Buffer{} },
}

buf := pool.Get().(*bytes.Buffer)
buf.Reset()
defer pool.Put(buf)
```

## Вопросы на собеседовании

### Q: В чём разница между стеком и кучей в Go?

**Ответ:** Стек — быстрая память для локальных переменных функции. Аллокация мгновенная (сдвиг указателя), освобождение автоматическое при возврате из функции, GC не нужен. Куча — общая память для объектов чей жизненный цикл выходит за рамки функции. Аллокация медленнее, освобождением управляет GC. Компилятор сам решает где жить объекту через escape analysis.

---

### Q: Почему горутины легче потоков ОС?

**Ответ:** Поток ОС имеет фиксированный стек 1-8 MB, выделяемый сразу. Горутина начинает с 2-8 KB и растёт по необходимости (copying stack). Поэтому 1000 потоков ОС = минимум 1 GB только под стеки. 1000 горутин = ~2-8 MB. Плюс переключение между горутинами дешевле — происходит в userspace без syscall.

---

### Q: Что такое escape analysis? Когда объект уходит на heap?

**Ответ:** Escape analysis — анализ компилятора, определяющий где жить объекту. На heap уходит если: возвращаем указатель на локальную переменную, переменная захвачена замыканием, передаём в интерфейс `any`, размер неизвестен на этапе компиляции, объект слишком большой. Проверяется через `go build -gcflags="-m"`. Цель — держать объекты на стеке как можно дольше, это быстрее и не нагружает GC.

---

### Q: Как Go увеличивает стек горутины?

**Ответ:** Go использует copying stack. При нехватке места аллоцируется новый стек в 2 раза больше, всё содержимое копируется, все указатели обновляются, старый стек освобождается. Раньше использовался segmented stack (цепочка сегментов), но он имел проблему hot split — частое пересечение границы сегмента вызывало постоянные аллокации/освобождения.
