# Каналы (Channels)

---

## Лекция

### Что такое канал и зачем он нужен

В конкурентном программировании главная проблема — безопасная передача данных между горутинами. Один из подходов — разделяемая память с мьютексами. Go предлагает другой подход, вдохновлённый теорией CSP (Communicating Sequential Processes):

> **"Don't communicate by sharing memory; share memory by communicating."**
> — Rob Pike

**Канал** — это типизированный, потокобезопасный канал связи между горутинами. Горутина кладёт значение в канал, другая горутина его читает. Синхронизация встроена в сам механизм.

---

### Внутреннее устройство канала

Под капотом канал — это структура `hchan` в рантайме:

```go
type hchan struct {
    qcount   uint           // количество элементов сейчас в буфере
    dataqsiz uint           // ёмкость кольцевого буфера (0 для небуферизованных)
    buf      unsafe.Pointer // кольцевой буфер (только для буферизованных)
    sendx    uint           // индекс куда писать следующий элемент
    recvx    uint           // индекс откуда читать следующий элемент
    recvq    waitq          // очередь горутин, ожидающих чтения (sudog linked list)
    sendq    waitq          // очередь горутин, ожидающих записи
    lock     mutex          // мьютекс для защиты полей структуры
}
```

**Небуферизованный канал:** `buf` = nil, `dataqsiz` = 0. Каждая запись блокируется до тех пор, пока не появится читатель — они "рандеву" (встречаются) в точке передачи.

**Буферизованный канал:** имеет кольцевой буфер. Запись не блокирует пока буфер не полон. Чтение не блокирует пока буфер не пуст.

---

### Виды каналов

```go
// Небуферизованный: синхронная передача, рандеву
ch := make(chan int)

// Буферизованный: асинхронная передача до заполнения буфера
ch := make(chan int, 10)

// Направленные: используются в сигнатурах функций для ограничения доступа
var send chan<- int = ch  // только запись
var recv <-chan int = ch  // только чтение
```

| | Небуферизованный | Буферизованный |
|---|---|---|
| Запись блокирует | всегда (пока нет читателя) | только если буфер полон |
| Чтение блокирует | всегда (пока нет писателя) | только если буфер пуст |
| Гарантия синхронизации | точка рандеву (гарантированная передача) | нет строгой гарантии |
| Применение | синхронизация горутин | буферизация, декаплинг |

**Когда какой использовать:**
- Небуферизованный — когда важно, что sender знает: receiver получил данные.
- Буферизованный — когда нужно развязать скорости sender/receiver или избежать блокировки на короткие промежутки.

### Аксиомы каналов — выучить наизусть

Это фундаментальные правила поведения каналов. На собеседовании могут спросить любое из них.

```go
// АКСИОМА 1: Запись в nil канал — блокируется НАВСЕГДА (deadlock)
var ch chan int
ch <- 1 // fatal error: all goroutines are asleep - deadlock!

// АКСИОМА 2: Чтение из nil канала — блокируется НАВСЕГДА (deadlock)
var ch chan int
<-ch // fatal error: all goroutines are asleep - deadlock!

// АКСИОМА 3: Запись в ЗАКРЫТЫЙ канал — ПАНИКА
ch := make(chan int)
close(ch)
ch <- 1 // panic: send on closed channel

// АКСИОМА 4: Чтение из закрытого канала — сначала буфер, потом zero value
ch := make(chan int, 1)
ch <- 42
close(ch)
v1, ok1 := <-ch // v1 = 42,  ok1 = true  (есть данные в буфере)
v2, ok2 := <-ch // v2 = 0,   ok2 = false (канал закрыт и пуст)

// АКСИОМА 5: Закрытие nil канала — ПАНИКА
var ch chan int
close(ch) // panic: close of nil channel

// АКСИОМА 6: Повторное закрытие канала — ПАНИКА
ch := make(chan int)
close(ch)
close(ch) // panic: close of closed channel
```

**Мнемоника:** паника возникает только при записи или закрытии — операции, которые "изменяют" канал в сторону "плохого состояния". Чтение из закрытого — безопасно, всегда возвращает значение.

---

### Как проверить, закрыт ли канал

```go
// Способ 1: двойное присваивание
v, ok := <-ch
if !ok {
    fmt.Println("канал закрыт, получили zero value")
}

// Способ 2: range — автоматически завершается когда канал закрыт и пуст
for v := range ch {
    fmt.Println(v)
}
// после выхода из цикла знаем: канал закрыт
```

**Нельзя проверить закрытость без чтения** — нет функции `isClosed(ch)`. Это сделано намеренно: такая проверка была бы race condition (между проверкой и использованием канал мог бы закрыться или нет).

### Select — мультиплексирование каналов

`select` — это `switch` для каналов. Позволяет ждать сразу несколько операций и выбрать первую готовую.

```go
select {
case msg := <-ch1:
    fmt.Println("получили из ch1:", msg)
case ch2 <- 42:
    fmt.Println("отправили в ch2")
case <-time.After(time.Second):
    fmt.Println("таймаут — никто не ответил за 1 секунду")
default:
    fmt.Println("неблокирующий select: ни один case не готов прямо сейчас")
}
```

**Важные свойства select:**

1. **Случайный выбор:** если несколько case готовы одновременно — выбирается **псевдослучайно**. Никакого приоритета по умолчанию нет.
2. **default делает select неблокирующим:** если ни один case не готов — выполняется default.
3. **`select {}` без case:** блокируется навсегда (полезно для main горутины).
4. **Nil канал в select:** case с nil-каналом никогда не выбирается — это как будто его нет.

---

### Приоритизация через двойной select

Если нужно, чтобы один канал имел приоритет над другим:

```go
// Приоритет: stopCh важнее dataCh
for {
    // Сначала проверяем только stopCh (неблокирующий)
    select {
    case <-stopCh:
        return
    default:
    }

    // Потом ждём любого из двух
    select {
    case <-stopCh:
        return
    case v := <-dataCh:
        process(v)
    }
}
```

### Nil канал как динамический "выключатель" case

```go
var secondary chan int // nil

select {
case v := <-primary:
    fmt.Println("primary:", v)
case v := <-secondary:   // никогда не выберется пока secondary == nil
    fmt.Println("secondary:", v)
}

// Включить secondary:
secondary = make(chan int, 1)
secondary <- 99
// Теперь case с secondary будет участвовать в select
```

Это мощная техника для динамического включения/отключения источников данных.

### Кто должен закрывать канал

**Правило:** закрывать канал должен **sender** (тот, кто пишет), потому что только он знает, когда данных больше не будет. Receiver не знает этого.

Если несколько sender-ов — нужна координация (например, через `sync.Once` или отдельный done-канал).

```go
// ✅ Один sender — он и закрывает
func producer(ch chan<- int) {
    defer close(ch) // закрываем когда всё отправлено
    for i := 0; i < 10; i++ {
        ch <- i
    }
}

// ✅ Несколько sender-ов — используем WaitGroup + горутину-closerR
func multiProducer(ch chan int, n int) {
    var wg sync.WaitGroup
    for i := 0; i < n; i++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            ch <- id
        }(i)
    }
    go func() {
        wg.Wait()
        close(ch) // закрываем только после завершения всех sender-ов
    }()
}
```

### Done-канал — сигнал остановки

Классический паттерн остановки горутин через закрытие канала:

```go
done := make(chan struct{}) // struct{} не занимает памяти

go func() {
    for {
        select {
        case <-done:
            fmt.Println("получили сигнал, останавливаюсь")
            return
        default:
            doWork()
        }
    }
}()

time.Sleep(time.Second)
close(done) // сигнализируем ВСЕМ читателям одновременно
// Закрытие канала — это broadcast: все, кто читает из него, получат сигнал
```

Преимущество закрытия над отправкой значения: `close(done)` разблокирует **все** горутины, ожидающие на этом канале. А `done <- struct{}{}` разбудит только одну.

## Примеры

### Пример 1: Небуферизованный канал — синхронизация

```go
package main

import (
    "fmt"
    "time"
)

func main() {
    ch := make(chan string)

    go func() {
        time.Sleep(time.Second)
        ch <- "результат готов" // блокируется пока main не прочитает
    }()

    fmt.Println("жду результат...")
    result := <-ch // блокируется пока горутина не отправит
    fmt.Println("получил:", result)
}
```

### Пример 2: Буферизованный канал — очередь задач

```go
package main

import "fmt"

func main() {
    tasks := make(chan int, 5) // буфер на 5 задач

    // Можем отправить до 5 задач без блокировки
    for i := 1; i <= 5; i++ {
        tasks <- i
        fmt.Printf("задача %d добавлена\n", i)
    }
    close(tasks)

    // Обрабатываем все задачи
    for task := range tasks {
        fmt.Printf("обрабатываю задачу %d\n", task)
    }
}
```

### Пример 3: Select с таймаутом

```go
package main

import (
    "fmt"
    "time"
)

func slowOperation() <-chan string {
    ch := make(chan string)
    go func() {
        time.Sleep(2 * time.Second) // долгая операция
        ch <- "готово"
    }()
    return ch
}

func main() {
    result := slowOperation()

    select {
    case v := <-result:
        fmt.Println("получили:", v)
    case <-time.After(1 * time.Second):
        fmt.Println("таймаут: операция слишком долго")
    }
}
```

### Пример 4: Неблокирующая отправка и чтение

```go
package main

import "fmt"

func tryRead(ch chan int) {
    select {
    case v := <-ch:
        fmt.Println("прочитали:", v)
    default:
        fmt.Println("канал пуст, не блокируемся")
    }
}

func tryWrite(ch chan int, v int) {
    select {
    case ch <- v:
        fmt.Println("записали:", v)
    default:
        fmt.Println("канал полон, не блокируемся")
    }
}

func main() {
    ch := make(chan int, 1)
    tryRead(ch)    // канал пуст, не блокируемся
    tryWrite(ch, 42) // записали: 42
    tryWrite(ch, 99) // канал полон, не блокируемся
    tryRead(ch)    // прочитали: 42
}
```

### Пример 5: Утечка горутин с каналами и исправление

```go
package main

import (
    "context"
    "fmt"
    "time"
)

// ❌ Утечка: producer будет заблокирован навсегда если consumer ушёл
func leakyProducer(ch chan<- int) {
    for i := 0; ; i++ {
        ch <- i // если никто не читает — горутина живёт вечно
    }
}

// ✅ Правильно: с context для отмены
func safeProducer(ctx context.Context, ch chan<- int) {
    for i := 0; ; i++ {
        select {
        case <-ctx.Done():
            fmt.Println("producer: остановился")
            return
        case ch <- i:
        }
    }
}

func main() {
    ctx, cancel := context.WithTimeout(context.Background(), 100*time.Millisecond)
    defer cancel()

    ch := make(chan int, 10)
    go safeProducer(ctx, ch)

    for v := range ch {
        fmt.Println("получили:", v)
        if v > 5 {
            cancel() // останавливаем producer
            break
        }
    }
}
```

## Вопросы с собеседования

**Q: Что выведет эта программа?**
```go
ch := make(chan int, 3)
ch <- 1
ch <- 2
ch <- 3
close(ch)
for v := range ch {
    fmt.Println(v)
}
```

Выведет `1`, `2`, `3`. Цикл `range` по закрытому буферизованному каналу сначала вычитывает все элементы из буфера, затем завершается. Паники нет — читать из закрытого канала можно.

---

**Q: Что произойдёт при записи в закрытый канал?**

Паника: `panic: send on closed channel`. Поэтому всегда нужно гарантировать, что закрытие происходит на стороне sender-а и только один раз.

---

**Q: Кто должен закрывать канал?**

По конвенции — **sender**. Receiver не знает когда данные закончатся. Если sender-ов несколько — координируем через `sync.WaitGroup`: отдельная горутина ждёт завершения всех sender-ов и затем закрывает канал.

---

**Q: Чем небуферизованный канал отличается от буферизованного?**

Небуферизованный: операции записи и чтения синхронизированы — запись блокируется до тех пор, пока не появится читатель, и наоборот. Гарантирует, что sender знает: receiver получил данные.

Буферизованный: запись не блокирует пока есть место в буфере. Позволяет развязать скорости sender и receiver.

---

**Q: Что такое select и как он выбирает между готовыми case?**

`select` ждёт сразу несколько канальных операций. Если несколько case готовы одновременно — выбирается **псевдослучайно** (равновероятно). Это предотвращает starvation. Если есть `default` — select неблокирующий: если ни один case не готов, выполняется default.

---

**Q: Что выведет?**
```go
var ch chan int
select {
case v := <-ch:
    fmt.Println(v)
default:
    fmt.Println("default")
}
```

Выведет `"default"`. Nil-канал в select никогда не готов (блокируется вечно), поэтому выполняется default.

---

**Q: Как сделать неблокирующую отправку в канал?**

Использовать select с default:
```go
select {
case ch <- value:
    // успешно отправили
default:
    // канал полон или нет читателей — не блокируемся
}
```

---

**Q: Почему нельзя написать функцию `isClosed(ch chan int) bool`?**

Потому что между проверкой и использованием результата другая горутина могла бы закрыть канал — это race condition. Безопасный способ проверки — только при чтении через двойное присваивание `v, ok := <-ch`.

---

**Q: Что такое "утечка горутины" в контексте каналов?**

Если горутина ожидает чтения/записи из канала, а операция никогда не произойдёт (канал заброшен, никто не пишет/читает) — горутина застывает в памяти навсегда. Решение: всегда использовать `context.Context` для управления временем жизни горутин, либо `select` с `done`-каналом.

---

**Q: В чём разница между `close(ch)` и отправкой sentinel-значения (например, -1)?**

`close(ch)` — это broadcast: все горутины, читающие из канала, получат нулевое значение с `ok=false`. Один `close` уведомляет всех читателей.

Sentinel-значение (например, `ch <- -1`) — это точечная отправка одному читателю. Чтобы уведомить N читателей, нужно N отправок. `close` элегантнее для сигнала остановки.
