# Горутины и планировщик Go (GMP)

---

## Лекция

### Что такое горутина и зачем она нужна

Когда программа запускается, ей нужно делать несколько вещей одновременно: обрабатывать запросы, читать из базы, писать в лог. В обычном мире для этого создают OS-потоки (threads). Но OS-поток — дорогой: он занимает 1–8 МБ стека и на его создание уходит ~10–100 мкс. Если у нас 10 000 запросов — создать 10 000 потоков нереально.

Go решает это через **горутины** — лёгкие единицы выполнения, которые управляются не операционной системой, а самим Go-рантаймом.

| | Горутина | OS Thread |
|---|---|---|
| Начальный стек | ~2–8 KB (растёт динамически до нескольких МБ) | 1–8 MB (фиксированный) |
| Создание | ~200 нс | ~10–100 мкс |
| Переключение | дешевле (user-space) | дороже (kernel syscall) |
| Планировщик | Go runtime | Операционная система |
| Типичный лимит | миллионы | тысячи |

Ключевой момент: **стек горутины растёт динамически**. Начинается с 2–8 КБ, а при необходимости Go выделяет новый, больший стек и копирует содержимое. Поэтому можно запустить миллион горутин без страха исчерпать память.

### GMP модель — сердце планировщика Go

Go использует **трёхуровневую модель M:N планирования**: M горутин выполняется на N OS-потоках.

```
G — Goroutine    (горутина: код + стек + состояние)
M — Machine      (OS thread: реальный поток операционной системы)
P — Processor    (логический процессор: очередь горутин + контекст выполнения)
```

Количество P определяется переменной `GOMAXPROCS` (по умолчанию = количество CPU).

```
[P1] → локальная очередь: [G1, G2, G3]
  └── M1 (OS thread) ──→ CPU core 0

[P2] → локальная очередь: [G4, G5]
  └── M2 (OS thread) ──→ CPU core 1

Глобальная очередь: [G6, G7, G8, ...]
```

**Почему именно такая структура?**

- P нужен, чтобы M не обращались к глобальной очереди при каждом переключении — это было бы узким местом (contention на мьютексе). Каждый P имеет локальную очередь до 256 горутин.
- M привязывается к P для выполнения. Без P — M не может выполнять горутины.
- При блокирующем syscall (например, файловый I/O) M "отсоединяется" от P, P находит другой свободный M (или создаёт новый) и продолжает работу. Заблокированный M потом либо возвращается, либо уходит в пул.

**Жизненный цикл горутины:**
1. `go func()` → создаётся G, помещается в локальную очередь P текущего потока
2. P берёт G из своей очереди и назначает на M для выполнения
3. G выполняется. При блокировке (channel, syscall, mutex) — G уходит в waiting, M ищет другую G
4. G завершается → её память освобождается

---

### Work Stealing — балансировка нагрузки

Представь: P1 обработал все горутины и стоит без дела, а P2 завален работой. Было бы расточительно.

Go решает это через **Work Stealing**:

1. Если локальная очередь P пуста → берём из **глобальной** очереди (batch, несколько штук)
2. Если глобальная тоже пуста → **крадём половину** горутин у случайного другого P

```
P1 [пусто]           P2 [G1, G2, G3, G4]
    │                     │
    └── крадёт ──────────→ │
                      P2 отдаёт [G3, G4]

После:
P1 [G3, G4]          P2 [G1, G2]
```

Это автоматически балансирует нагрузку без явного управления со стороны разработчика.

### Кооперативное vs Вытесняющее планирование

**До Go 1.14: только кооперативное планирование**

Горутина сама уступала управление в определённых "точках вытеснения":
- вызов функции (проверка стека)
- операции с каналами
- системные вызовы
- `runtime.Gosched()`

Проблема: горутина с пустым циклом `for {}` могла "захватить" поток и не давать другим работать.

**С Go 1.14: асинхронное вытеснение (async preemption)**

Рантайм отправляет сигнал `SIGURG` каждые ~10 мс. Сигнал-хэндлер безопасно прерывает горутину и возвращает управление планировщику. Теперь ни одна горутина не может монополизировать поток.

```go
// До Go 1.14 это могло заблокировать все остальные горутины на этом P:
go func() {
    for {
        // пустой цикл без точек вытеснения
    }
}()
// С Go 1.14 — планировщик прервёт эту горутину через ~10 мс
```

### GOMAXPROCS

`GOMAXPROCS` — количество логических процессоров P, то есть максимальное количество горутин, выполняющихся **параллельно** в один момент.

```go
import "runtime"

procs := runtime.GOMAXPROCS(0) // 0 = не менять, только вернуть текущее
runtime.GOMAXPROCS(4)          // установить явно

// Или через переменную окружения при запуске:
// GOMAXPROCS=4 ./myapp
```

По умолчанию = `runtime.NumCPU()`. На продакшн-серверах с 32 CPU будет 32 параллельных потока.

**Go 1.25+: контейнерное awareness.** В Linux GOMAXPROCS автоматически учитывает cgroup CPU bandwidth limits — если контейнеру выделено 2 CPU из 32, GOMAXPROCS будет 2, а не 32. Это предотвращает избыточный параллелизм в K8s/Docker. Можно отключить через `GODEBUG=containermaxprocs=0`.

**Важно понимать:** GOMAXPROCS ограничивает параллелизм, но не конкурентность. Горутин может быть миллион, а выполняться одновременно — только GOMAXPROCS штук.

### Классические ловушки

**Ловушка 1: захват переменной цикла замыканием**

> **Go 1.22+:** эта ловушка **устранена** — каждая итерация цикла создаёт новую переменную. Код ниже корректно выведет `0, 1, 2` начиная с Go 1.22. На собеседовании важно знать оба поведения.

```go
// ❌ Go 1.21 и ниже: все горутины захватывают одну и ту же переменную i
for i := 0; i < 3; i++ {
    go func() {
        fmt.Println(i) // скорее всего напечатает 3, 3, 3
    }()
}
// К моменту выполнения горутин цикл уже завершился, i == 3

// ✅ Go 1.22+: каждая итерация — новая переменная, ловушки нет
// Тот же код теперь корректен автоматически
```

```go
// ✅ Явный workaround (нужен для Go 1.21 и ниже, в 1.22+ избыточен)
for i := 0; i < 3; i++ {
    go func(id int) {
        fmt.Println(id) // 0, 1, 2 в любом порядке
    }(i)
}
```

**Ловушка 2: утечка горутин**

Горутина не завершается → продолжает занимать память и ресурсы → утечка.

```go
// ❌ Утечка: горутина заблокирована на канале навсегда
func leak() {
    ch := make(chan int)
    go func() {
        val := <-ch // никто не пишет — горутина живёт вечно
        fmt.Println(val)
    }()
    // ch выходит из области видимости, но горутина не завершается
}

// ✅ Используем context для управления временем жизни
func noLeak(ctx context.Context) {
    ch := make(chan int)
    go func() {
        select {
        case val := <-ch:
            fmt.Println(val)
        case <-ctx.Done():
            return // горутина завершается при отмене контекста
        }
    }()
}
```

**Правило:** каждая запущенная горутина должна иметь чёткий способ завершения.


## Примеры

### Пример 1: WaitGroup для ожидания горутин

```go
package main

import (
    "fmt"
    "sync"
)

func main() {
    var wg sync.WaitGroup

    for i := 0; i < 5; i++ {
        wg.Add(1)            // добавляем ПЕРЕД запуском горутины
        go func(id int) {
            defer wg.Done() // уменьшаем счётчик при завершении
            fmt.Printf("worker %d done\n", id)
        }(i)
    }

    wg.Wait() // блокируемся пока все не завершатся
    fmt.Println("все воркеры завершили работу")
}
```

### Пример 2: runtime.Gosched() — явная уступка

```go
package main

import (
    "fmt"
    "runtime"
    "sync"
)

func main() {
    runtime.GOMAXPROCS(1) // один P для наглядности

    var wg sync.WaitGroup
    wg.Add(2)

    go func() {
        defer wg.Done()
        for i := 0; i < 3; i++ {
            fmt.Println("горутина A:", i)
            runtime.Gosched() // явно уступаем — дадим B поработать
        }
    }()

    go func() {
        defer wg.Done()
        for i := 0; i < 3; i++ {
            fmt.Println("горутина B:", i)
            runtime.Gosched()
        }
    }()

    wg.Wait()
}
// Вывод будет чередоваться: A:0, B:0, A:1, B:1, ...
```

### Пример 3: Детектор гонки данных

```go
// Запуск: go run -race main.go

package main

import (
    "fmt"
    "sync"
)

func main() {
    var counter int
    var wg sync.WaitGroup

    for i := 0; i < 1000; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            counter++ // ❌ DATA RACE: несколько горутин пишут без синхронизации
        }()
    }

    wg.Wait()
    fmt.Println(counter) // результат непредсказуем: может быть меньше 1000
}
```

### Пример 4: Graceful shutdown горутин через context

```go
package main

import (
    "context"
    "fmt"
    "time"
)

func worker(ctx context.Context, id int) {
    for {
        select {
        case <-ctx.Done():
            fmt.Printf("worker %d: получил сигнал остановки\n", id)
            return
        default:
            fmt.Printf("worker %d: работаю...\n", id)
            time.Sleep(300 * time.Millisecond)
        }
    }
}

func main() {
    ctx, cancel := context.WithTimeout(context.Background(), time.Second)
    defer cancel()

    for i := 1; i <= 3; i++ {
        go worker(ctx, i)
    }

    <-ctx.Done()
    time.Sleep(100 * time.Millisecond) // даём горутинам завершиться
    fmt.Println("все воркеры остановлены")
}
```

## Вопросы с собеседования

**Q: Что такое горутина? Чем отличается от OS-потока?**

Горутина — лёгкий поток выполнения, управляемый Go-рантаймом, а не ОС. Начальный стек ~2–8 КБ против 1–8 МБ у OS-потока. Создание занимает ~200 нс против ~10–100 мкс. Планирование происходит в user-space без системных вызовов, поэтому переключение дешевле. Можно запустить миллионы горутин, OS-потоков — только тысячи.

---

**Q: Объясните GMP модель.**

- **G (Goroutine)** — сама горутина: код, стек, состояние.
- **M (Machine)** — OS-поток, реально выполняет код на CPU.
- **P (Processor)** — логический процессор, хранит локальную очередь горутин и связывает M с G. Количество P = GOMAXPROCS.

M выполняет горутины только при наличии P. При блокирующем syscall M отсоединяется от P, P находит другой M, и работа продолжается. Это позволяет эффективно использовать CPU даже при большом количестве I/O-блокировок.

---

**Q: Что такое Work Stealing?**

Механизм балансировки: если локальная очередь P пуста, он сначала берёт горутины из глобальной очереди, а если она тоже пуста — крадёт половину горутин у случайного другого P. Это автоматически выравнивает нагрузку между процессорами.

---

**Q: Сколько OS-потоков создаст Go?**

Минимум один M на каждый P. При блокирующих syscall-ах могут создаваться дополнительные M. Максимальное количество M ограничено `runtime/debug.SetMaxThreads` (по умолчанию 10 000).

---

**Q: Что такое `runtime.Gosched()`?**

Явная уступка управления планировщику. Текущая горутина переходит в конец очереди и даёт выполниться другим. Аналог `Thread.yield()` в Java. Используется редко — обычно для тестирования или в tight loop без других точек вытеснения.

---

**Q: Чем конкурентность отличается от параллелизма?**

- **Конкурентность** (concurrency) — это структура программы: несколько задач могут прогрессировать, чередуясь (даже на одном ядре).
- **Параллелизм** (parallelism) — физическое одновременное выполнение на нескольких ядрах.

Go предоставляет инструменты конкурентности (горутины, каналы). Параллелизм определяется GOMAXPROCS. При GOMAXPROCS=1 программа конкурентна, но не параллельна.

---

**Q: Как найти утечки горутин?**

1. `runtime.NumGoroutine()` — посмотреть текущее количество
2. `net/http/pprof` — эндпоинт `/debug/pprof/goroutine` покажет стек каждой горутины
3. Библиотека `goleak` (uber-go/goleak) — в тестах детектирует горутины, не завершившиеся после теста
4. **Go 1.26+:** экспериментальный pprof-профиль `goroutineleak` — находит постоянно заблокированные горутины через анализ достижимости GC (`GOEXPERIMENT=goroutineleakprofile`, эндпоинт `/debug/pprof/goroutineleak`)

---

**Q: Что произойдёт если GOMAXPROCS=1 и одна горутина выполняет пустой цикл `for {}`?**

До Go 1.14 — остальные горутины не получат CPU (нет точек вытеснения). С Go 1.14+ — async preemption через SIGURG прервёт цикл через ~10 мс и планировщик переключится на другие горутины.

---

**Q: Почему нельзя полагаться на порядок выполнения горутин?**

Планировщик не гарантирует порядок. Горутины могут выполняться в любой последовательности в зависимости от состояния очередей, Work Stealing, наличия свободных M/P. Для упорядочивания нужны явные механизмы синхронизации: каналы, мьютексы, WaitGroup.
