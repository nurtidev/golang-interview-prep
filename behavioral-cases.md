### Кейс 1: "Дублирующиеся транзакции в продакшне" (eMoney)

**Вопрос-триггер:** *"Расскажи о сложной баге которую ты находил и фиксил"* /
*"Опиши ситуацию где ты принял важное техническое решение"*

**Situation:**
В eMoney мы обрабатывали платежи через внешних провайдеров. Провайдеры иногда не отвечали в срок, и наш сервис делал retry. После одного деплоя мы заметили в Grafana аномалию: ~5% транзакций дублировались — деньги списывались дважды.

**Task:**
Найти причину и устранить без остановки сервиса. Данные клиентов — деньги, каждая минута простоя или некорректной работы критична.

**Action:**
Первое что я сделал — включил подробное логирование на входе в сервис платежей и стал смотреть паттерн. Оказалось: при timeout от провайдера (5 секунд) мы делали retry, но провайдер на самом деле успевал обработать запрос — просто ответ приходил позже. Второй запрос создавал вторую транзакцию.

Решение было многоуровневым:

1. **Idempotency key на уровне БД**: добавил `UNIQUE(idempotency_key)` в таблицу payments. Клиент генерирует UUID и прокидывает в заголовке, мы его сохраняем. При дубле — `INSERT ... ON CONFLICT DO NOTHING` + возвращаем уже сохранённый результат.

2. **Redis для быстрой проверки**: перед INSERT проверяем Redis (`GET idempotency:{key}`). Если есть — отвечаем немедленно без похода в PostgreSQL.

3. **Retry только для idempotent операций**: GET-запросы к провайдеру ретраим всегда. POST (создание транзакции) — только если провайдер вернул `5xx` или timeout. Если `2xx` — не ретраим никогда.

4. **Тест**: написал integration test который симулировал медленный провайдер (sleep + задержка ответа) и проверял что при 100 параллельных запросах с одним ключом — ровно 1 транзакция в БД.

**Result:**
Количество дублирующихся транзакций снизилось на ~35% в первую неделю. После полного роллаута — до нуля. Паттерн стал стандартом для всех новых платёжных эндпоинтов в команде.

### Кейс 2: "Сервис падал под нагрузкой" (Sergek)

**Вопрос-триггер:** *"Расскажи о проблеме с производительностью которую ты решал"* /
*"Как ты оптимизировал систему?"*

**Situation:**
В Sergek мы обрабатывали события с городских камер (~200k событий/сутки). В часы пик (утро/вечер) сервис начинал деградировать: latency росла с 50ms до 3-4 секунд, часть событий терялась.

**Task:**
Найти bottleneck и обеспечить стабильную обработку даже в пиковые часы.

**Action:**
Подключил pprof на staging-окружение и воспроизвёл нагрузку через нагрузочный тест. CPU профиль показал что 60%+ времени — это `encoding/json.Marshal` и `db.Exec` в tight loop.

Причина оказалась в архитектуре: мы обрабатывали события **поштучно** — один Kafka consumer читал одно сообщение, сразу маршалил, сразу делал INSERT в ClickHouse. Это создавало огромное количество мелких операций.

Решение — **micro-batching**:

```go
// БЫЛО: поштучная обработка
for msg := range consumer.Messages() {
    event := parseEvent(msg)
    db.Insert(event) // 1 INSERT = 1 roundtrip
}

// СТАЛО: пакетная вставка
var buf []Event
ticker := time.NewTicker(100 * time.Millisecond)
for {
    select {
    case msg := <-consumer.Messages():
        buf = append(buf, parseEvent(msg))
        if len(buf) >= 500 { // flush по размеру
            db.BulkInsert(buf)
            buf = buf[:0]
        }
    case <-ticker.C: // flush по времени
        if len(buf) > 0 {
            db.BulkInsert(buf)
            buf = buf[:0]
        }
    }
}
```

Также мигрировали с PostgreSQL на ClickHouse для хранения событий — он оптимизирован для bulk insert и аналитических запросов.

**Result:**
- Throughput вырос в 10x
- Latency в пиковые часы: 3-4s → ~80ms
- Скорость аналитических запросов выросла в 4-5x (ClickHouse vs PostgreSQL для этих паттернов)

### Кейс 3: "Утечка горутин которую долго не замечали" (Darwin Tech)

**Вопрос-триггер:** *"Был ли у тебя опыт с concurrency проблемами?"* /
*"Расскажи о нетривиальном баге"*

**Situation:**
В Darwin Tech мы строили микросервисную торговую платформу. Один из сервисов (order execution) медленно рос по памяти — примерно на 50MB в час. Через 24 часа — OOM и рестарт. Это было маскировано автоматическим рестартом в Kubernetes, так что долго никто не замечал.

**Task:**
Найти причину роста памяти.

**Action:**
Первая гипотеза — утечка памяти в БД (connections не закрываются). Проверил через `db.Stats()` — всё в норме. Тогда снял heap профиль через pprof:

```bash
go tool pprof http://localhost:6060/debug/pprof/goroutine?debug=2
```

Увидел что количество горутин монотонно растёт: ~200 при старте, ~8000 через 6 часов. Все "висят" в одном месте:
```
goroutine [chan receive, 4 hours]:
order/execution.(*Processor).waitForExternalConfirmation(...)
```

Нашёл код:
```go
// ПЛОХО: горутина висит вечно если внешняя система не ответила
func (p *Processor) waitForExternalConfirmation(orderID string) {
    confirmCh := make(chan bool)
    go func() {
        result := p.externalSystem.WaitConfirmation(orderID) // блокирующий вызов
        confirmCh <- result
    }()
    confirmed := <-confirmCh // если externalSystem завис — горутина не освободится НИКОГДА
}
```

Внешняя система иногда зависала без ответа. Горутина висела вечно, а с ней — вся память.

Фикс:
```go
func (p *Processor) waitForExternalConfirmation(ctx context.Context, orderID string) error {
    ctx, cancel := context.WithTimeout(ctx, 30*time.Second)
    defer cancel()
    
    resultCh := make(chan error, 1) // буфер! горутина не зависнет при записи
    go func() {
        resultCh <- p.externalSystem.WaitConfirmation(ctx, orderID)
    }()
    
    select {
    case err := <-resultCh:
        return err
    case <-ctx.Done():
        return fmt.Errorf("confirmation timeout for order %s", orderID)
    }
}
```

**Result:**
После деплоя — горутины стабилизировались на ~200. Память перестала расти. Добавили алерт на количество горутин (`goroutines_count > 1000`) в Prometheus/Grafana.

### Кейс 4: "Конфликт в команде по архитектурному решению"

**Вопрос-триггер:** *"Расскажи о конфликте в команде"* / *"Как ты убеждал коллег?"*

**Situation:**
В Darwin Tech встал вопрос: как синхронизировать данные между order service и balance service. Тимлид предлагал синхронный REST-вызов (проще, быстрее написать). Я настаивал на Kafka (асинхронно, надёжнее).

**Task:**
Принять правильное техническое решение и не создать конфликт в команде.

**Action:**
Вместо споров я предложил смоделировать оба варианта на примере:

**Вариант REST (синхронный):**
```
Order Service → POST /balance/deduct → Balance Service
               ↑
         Что если timeout? 
         Order создан, но деньги не списаны.
         Retry может задвоить списание.
         Distributed transaction — очень сложно.
```

**Вариант Kafka:**
```
Order Service → publishes OrderCreated → Kafka
                                          ↓
                              Balance Service consumes
                              + idempotency key
                              + at-least-once delivery
```

Я написал простой документ (1 страница) с comparison + предложил провести PoC — за 2 дня сделал working prototype с Kafka включая consumer с идемпотентностью.

Тимлид увидел что prototype работает и проблема двойного списания решена. Согласился с Kafka.

**Result:**
Внедрили Kafka. За следующие полгода сервис ни разу не имел проблем с рассинхронизацией балансов. Тимлид потом сам стал использовать этот паттерн в других местах. Я из этого вынес: технические споры лучше решать кодом и цифрами, а не словами.

### Кейс 5: "Как работал со внешним API которое было ненадёжным" (eMoney)

**Вопрос-триггер:** *"Как ты работаешь с внешними зависимостями?"* /
*"Расскажи о работе с третьей стороной"*

**Situation:**
В eMoney мы интегрировались с несколькими платёжными провайдерами. Один из них (назову Provider X) имел нестабильный API: периодические timeout, иногда 503 на несколько минут.

**Task:**
Обеспечить что наш сервис остаётся стабильным даже когда Provider X деградирует.

**Action:**
Реализовал несколько слоёв защиты:

1. **Circuit Breaker**: если 5 запросов подряд завершились ошибкой — "размыкаем цепь" на 30 секунд. В течение этого времени сразу возвращаем ошибку (без ожидания timeout). Через 30 секунд — пробный запрос, если успешен — снова работаем.

2. **Retry с exponential backoff**: при timeout ретраим 3 раза с задержками 100ms → 200ms → 400ms. Важно: только для идемпотентных операций (проверка статуса), никогда — для создания транзакции.

3. **Timeout на каждый запрос**: через `context.WithTimeout(ctx, 3*time.Second)`. Без этого одна "зависшая" горутина могла держать connection pool.

4. **Fallback для некритичных операций**: проверка статуса транзакции — если провайдер недоступен, берём последнее известное состояние из нашей БД и помечаем `status: pending_verification`. Background worker потом досинхронизирует.

**Result:**
Когда Provider X имел инцидент (2 часа недоступности), наш сервис продолжал работать: операции шли в `pending` статусе. После восстановления провайдера — background reconciliation восстановил все статусы. Zero data loss, SLA не нарушен.

**"Назови свою самую большую ошибку"**
