# Нормализация баз данных

## Зачем нормализация

Без нормализации возникают аномалии:

| Аномалия | Пример |
|---|---|
| **Вставки** | Нельзя добавить водителя без поездки если driver_id — часть PK |
| **Удаления** | Удалили последнюю поездку водителя — потеряли информацию о нём |
| **Обновления** | Водитель сменил телефон — нужно обновить 1000 строк в таблице поездок |

Нормализация устраняет эти аномалии через устранение избыточности.

---

## 1NF — Первая нормальная форма

**Правило:** каждая ячейка содержит атомарное (неделимое) значение. Нет повторяющихся групп, нет массивов в колонке.

### Нарушение 1NF

```
rides:
| ride_id | passenger | phones              | cities          |
|---------|-----------|---------------------|-----------------|
| 1       | Иван      | +7777, +7888        | Москва, Питер   |
| 2       | Мария     | +7999               | Казань          |
```

Проблемы:
- Нельзя выбрать `WHERE phone = '+7777'` без LIKE или парсинга
- Нельзя нормально проиндексировать
- Непонятная длина поля

### После 1NF

```sql
-- Разбиваем на отдельные строки / таблицы
CREATE TABLE user_phones (
    user_id BIGINT NOT NULL REFERENCES users(id),
    phone   VARCHAR(20) NOT NULL,
    PRIMARY KEY (user_id, phone)
);
```

**Признаки нарушения 1NF:**
- Колонки типа `tags VARCHAR` с `"tag1,tag2,tag3"`
- Колонки `phone1`, `phone2`, `phone3` (повторяющиеся группы)
- JSON-массив там где нужна фильтрация/поиск

---

## 2NF — Вторая нормальная форма

**Правило:** таблица в 1NF + каждая неключевая колонка зависит от **всего** первичного ключа, а не от его части.

Актуально только для таблиц с **составным PK**.

### Нарушение 2NF

```
order_items:
| order_id | product_id | quantity | product_name | product_price |
|----------|------------|----------|--------------|---------------|
| 1        | 101        | 2        | Наушники     | 5000          |
| 1        | 102        | 1        | Кабель       | 500           |
| 2        | 101        | 1        | Наушники     | 5000          |
```

PK = `(order_id, product_id)`, но:
- `product_name` зависит только от `product_id` — **частичная зависимость**
- `product_price` зависит только от `product_id` — **частичная зависимость**

Если изменится цена наушников — надо обновить все строки где `product_id = 101`.

### После 2NF

```sql
CREATE TABLE products (
    id    BIGINT PRIMARY KEY,
    name  VARCHAR(100) NOT NULL,
    price NUMERIC(10,2) NOT NULL
);

CREATE TABLE order_items (
    order_id   BIGINT NOT NULL REFERENCES orders(id),
    product_id BIGINT NOT NULL REFERENCES products(id),
    quantity   INT    NOT NULL,
    -- price_at_purchase — фиксируем цену на момент заказа (это нормально, это факт заказа)
    unit_price NUMERIC(10,2) NOT NULL,
    PRIMARY KEY (order_id, product_id)
);
```

## 3NF — Третья нормальная форма

**Правило:** таблица в 2NF + нет транзитивных зависимостей. Неключевая колонка не должна зависеть от другой неключевой колонки.

### Нарушение 3NF

```
employees:
| emp_id | name  | dept_id | dept_name  | dept_location |
|--------|-------|---------|------------|---------------|
| 1      | Иван  | 10      | Инженеры   | Москва        |
| 2      | Мария | 10      | Инженеры   | Москва        |
| 3      | Пётр  | 20      | Маркетинг  | Питер         |
```

PK = `emp_id`, но:
- `dept_name` зависит от `dept_id` (не от `emp_id`) — **транзитивная зависимость**
- `dept_location` зависит от `dept_id` — **транзитивная зависимость**

Цепочка: `emp_id → dept_id → dept_name` — нарушение 3NF.

### После 3NF

```sql
CREATE TABLE departments (
    id       BIGINT PRIMARY KEY,
    name     VARCHAR(100) NOT NULL,
    location VARCHAR(100) NOT NULL
);

CREATE TABLE employees (
    id      BIGINT PRIMARY KEY,
    name    VARCHAR(100) NOT NULL,
    dept_id BIGINT NOT NULL REFERENCES departments(id)
);
```

## BCNF — Нормальная форма Бойса-Кодда

**Правило:** усиленная 3NF. Для каждой функциональной зависимости `X → Y`, X должен быть суперключом.

На практике встречается редко. Пример нарушения: таблица с несколькими перекрывающимися кандидатными ключами.

```
-- Преподаватель ведёт только один предмет, предмет может вести несколько преподавателей
-- Студент записан к одному преподавателю по каждому предмету
course_assignments:
| student | subject | teacher  |
|---------|---------|----------|
| Иван    | Math    | Иванов   |
| Мария   | Math    | Иванов   |
| Иван    | Physics | Петров   |

-- Зависимости:
-- (student, subject) → teacher  (студент + предмет определяют учителя)
-- teacher → subject              (учитель ведёт только один предмет — нарушение BCNF!)
-- teacher не является суперключом, но определяет subject
```

На практике такие ситуации редки и часто решаются пересмотром модели, а не механическим приведением к BCNF.

---

## 4NF и 5NF (кратко)

**4NF:** нет многозначных зависимостей. Если один факт (например, навыки сотрудника) и другой факт (проекты сотрудника) независимы — хранить в разных таблицах, а не в одной.

**5NF (PJ/NF):** нет аномалий при декомпозиции и соединении. Теоретическая форма, на практике почти не применяется.

---

## Практическое правило

```
1NF → атомарные значения, нет повторяющихся групп
2NF → нет частичных зависимостей от части составного PK
3NF → нет транзитивных зависимостей между неключевыми полями
BCNF → каждый детерминант является суперключом
```

**На собесе достаточно уверенно знать 1NF–3NF** и понимать суть: каждый факт хранится ровно в одном месте.

## Денормализация — осознанный трейдофф

Нормализация — правило по умолчанию. Денормализация — когда есть измеримая проблема.

```sql
-- Нормализовано: JOIN на каждый запрос
SELECT r.id, u.name, u.phone
FROM rides r
JOIN users u ON u.id = r.passenger_id;

-- Денормализовано: passenger_name прямо в rides (для read-heavy таблицы)
ALTER TABLE rides ADD COLUMN passenger_name VARCHAR(100);
-- Цена: при смене имени нужно обновить все поездки пользователя
```

**Когда денормализовать:**
- JOIN по нескольким большим таблицам стал узким местом (есть метрики)
- Read >> Write, данные меняются редко
- Хранишь исторический снапшот (цена товара на момент заказа — это правильно!)

**Инструменты вместо денормализации:** Materialized Views, Redis-кэш, CQRS с отдельной read-моделью.

## Частые вопросы на интервью (Q&A)

**Q: Что такое 1NF, 2NF, 3NF?**

1NF — атомарные значения, нет массивов/списков в ячейке. 2NF — нет частичных зависимостей от части составного ключа (каждое поле зависит от всего PK). 3NF — нет транзитивных зависимостей между неключевыми полями (A → B → C где B и C — не ключи — нарушение).

**Q: Что такое транзитивная зависимость?**

Когда неключевое поле зависит не от первичного ключа напрямую, а через другое неключевое поле. Пример: `employee_id → department_id → department_city` — city зависит от department_id, а не от employee_id. Решение — вынести в отдельную таблицу departments.

**Q: Всегда ли нужна нормализация?**

Нет. В OLAP/аналитических системах (ClickHouse, Redshift) намеренно денормализуют: JOIN дорог на миллиардах строк, поэтому дублируют поля ради скорости агрегаций. В OLTP (транзакционные системы) нормализация по умолчанию — защищает от аномалий обновления.

**Q: Чем отличается BCNF от 3NF?**

3NF допускает, что неключевое поле может быть частью кандидатного ключа и при этом зависеть от другого кандидатного ключа. BCNF это запрещает — каждый детерминант обязан быть суперключом. На практике большинство таблиц в 3NF автоматически в BCNF.
